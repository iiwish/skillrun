# v0.5.6 Headless Consumer Contract

## 一句话判断

v0.5.6 不应该急着进入 Desktop UI；它应该先定义 SkillRun Core 给 Desktop、Router 和自动化消费者使用的 headless contract，让后续 Tauri 项目消费稳定接口，而不是读取内部目录或反解析 MCP 文本。

## 背景

v0.5.5 已经把 Consumer Mode 的核心执行边界统一到静态 Manifest：

- execution 前验证 Manifest。
- MCP exposure 前验证 Manifest。
- `.skr` pack 前验证 Manifest。
- registry / switchboard 作为本地 inventory 和 enable state。
- `inspect/check/doctor --json` 已经给 Desktop 提供基础只读面。

但 v0.6 Consumer Era 还缺一个前置层：

> 一个不带 UI 的本地消费者控制面合同。

没有这层合同，Desktop 会被迫做三件不该做的事：

- 直接读取 `.skillrun/` 内部目录结构。
- 直接解析 CLI human text 或 MCP text content。
- 在 UI 项目里重新定义 Manifest、registry、switchboard 或 run record 语义。

## 目标

v0.5.6 的目标是冻结最小 headless consumer backend contract。

它应回答：

- 当前本机有哪些 capsule？
- 哪些 capsule 已启用？
- 哪些 capsule 不能运行，原因是什么？
- 当前 MCP client 是否已经挂载 SkillRun Router？
- 启用的 capsule 会暴露哪些 Manifest-derived tools？
- 最近执行过哪些 run，输出、错误、artifact 和耗时是什么？
- UI / automation 应该消费哪些稳定 JSON，而不是读哪些内部文件？

## 非目标

v0.5.6 不做：

- Tauri / Desktop UI。
- marketplace。
- signed package trust。
- dependency vendoring。
- OS sandbox。
- 自动安装 Python、Node、Pydantic 或平台 SDK。
- 通用 Agent orchestration。
- 完整 OpenAPI-to-skill generator。
- 多用户权限系统。

## 核心原则

### P001 Desktop 是 consumer，不是 runtime owner

Desktop 不应该直接执行 action、直接 import action source、直接解析 Manifest YAML 或直接读取 `.skillrun/runs` 作为稳定 API。

Desktop 应只消费：

- CLI JSON。
- 未来 local daemon API。
- GitHub Release / docs 中明确承诺的 JSON contract。

### P002 Router 是 MCP 暴露入口，`.skr` 不是

一键挂载不应该把 `.skr` 直接挂到 Claude Desktop、Cursor 或其他 MCP client。

`.skr` 是 source + Manifest archive；运行入口应是本地 SkillRun Router / Gateway。

### P003 Registry 是 inventory，不是 trust store

`registry add` 只表示“本机知道这个 capsule 路径”。

它不表示：

- trusted。
- installed。
- enabled。
- runnable。
- reviewed。

真正能否暴露给 MCP client，必须由 switchboard enable state 和 readiness gate 决定。

### P004 Run evidence 是一等数据面

Envelope Explorer 不能从 MCP client text 反推出审计事实。

它必须消费 SkillRun run records / envelopes / artifacts 的结构化 JSON。

### P005 Headless contract 先于 UI polish

如果某个 UI 页面需要读取一个事实，但 Core 没有稳定 JSON surface，就先补 Core contract，不要让 Desktop 项目自己猜。

## 当前可复用 surface

v0.5.6 可以直接复用这些已存在 surface：

| Surface | 用途 | 当前判断 |
| --- | --- | --- |
| `skillrun inspect --json --cwd <capsule>` | 读取单个 capsule 的 Manifest-derived 摘要 | 可作为 Desktop detail 输入 |
| `skillrun check --json --cwd <capsule>` | 读取 readiness、dependency、source hash 状态 | 可作为 enable 前 gate 输入 |
| `skillrun doctor --json --cwd <capsule>` | 读取诊断与恢复建议 | 可作为 UI 修复建议输入 |
| `skillrun registry list` / JSON surface | 读取本地 capsule inventory | 可作为 switchboard 列表输入 |
| `skillrun switchboard list` / JSON surface | 读取 enable state | 可作为 tool exposure 控制输入 |
| `skillrun test/run` output envelope | 读取单次执行结果 | 可作为 run detail 输入 |
| `.skr` package | source + Manifest 分发物 | 只能作为 import artifact，不能作为 runtime entry |

如果某个 surface 目前只有 human output，v0.5.6 应优先补 `--json` 或明确不承诺 Desktop 消费。

## 需要新增或冻结的 contract

### C001 Capsule Inventory Contract

回答：

> 当前本机有哪些 capsule，它们分别是什么状态？

建议稳定字段：

```json
{
  "command": "consumer inventory",
  "schema_version": "consumer.inventory.v1",
  "version": 1,
  "registry_path": "<path>",
  "capsules": [
    {
      "id": "refund",
      "path": "<capsule-path>",
      "source_type": "local_path",
      "enabled": true,
      "registered_at": "<timestamp>",
      "manifest": {
        "path": ".skillrun/manifest.generated.yaml",
        "present": true,
        "freshness": "fresh"
      },
      "skill": {
        "name": "refund"
      },
      "runtime": {
        "adapter": "python",
        "entrypoint": "action.py"
      },
      "tool": {
        "name": "refund"
      },
      "readiness": {
        "ok": true,
        "status": "ok",
        "next_step": "none"
      }
    }
  ]
}
```

约束：

- 单个坏 Manifest 不得拖垮整个 list。
- `enabled=true` 不等于当前 runnable。
- `readiness.ok=false` 时 UI 必须展示原因，而不是隐藏条目。

### C002 Switchboard Exposure Contract

回答：

> 哪些 enabled capsules 会被 Router 暴露成 MCP tools？

建议稳定字段：

```json
{
  "command": "consumer exposure",
  "schema_version": "consumer.exposure.v1",
  "registry_path": "<path>",
  "tools": [
    {
      "capsule_id": "refund",
      "tool_name": "refund",
      "enabled": true,
      "exposed": true,
      "readiness_status": "ok",
      "manifest_hash": "<sha256>"
    }
  ]
}
```

约束：

- `enabled=false` 的 capsule 不暴露。
- `readiness.ok=false` 的 capsule 不暴露，即使 registry 里 enabled state 仍为 true。
- 暴露工具名必须来自 Manifest-derived tool contract。

### C003 Router Status Contract

回答：

> SkillRun Router 是否运行，暴露了什么，挂载到哪些 MCP client？

v0.5.6 可以先定义合同，不急实现 daemon。

建议稳定字段：

```json
{
  "schema_version": "consumer.router_status.v1",
  "router": {
    "running": false,
    "mode": "stdio",
    "command": "skillrun router serve --mcp",
    "pid": null
  },
  "exposure": {
    "enabled_capsules": 0,
    "exposed_tools": 0
  },
  "clients": [
    {
      "id": "claude-desktop",
      "name": "Claude Desktop",
      "detected": true,
      "mounted": false,
      "config_path": "<path>",
      "last_backup_path": null
    }
  ]
}
```

约束：

- status 不得修改 MCP client 配置。
- status 不得启动 action。
- status 可以读取 client config，但必须区分 detected / mounted / unsupported。

### C004 Mount Plan Contract

回答：

> 如果用户点击一键挂载，会改哪些文件？

挂载必须先生成 plan，再 apply。

建议 plan 字段：

```json
{
  "command": "consumer mount plan",
  "schema_version": "consumer.mount_plan.v1",
  "client": {
    "id": "claude-desktop",
    "name": "Claude Desktop",
    "supported": true,
    "detected": true
  },
  "operation": "install_or_update_router",
  "config": {
    "path": "<path>",
    "path_source": "detected",
    "exists": true,
    "parseable": true
  },
  "backup": {
    "path": "<path>.skillrun.bak",
    "required_before_apply": true
  },
  "router": {
    "server_name": "skillrun",
    "command": "skillrun",
    "args": ["router", "serve", "--mcp"]
  },
  "changes": [
    {
      "kind": "upsert_mcp_server",
      "server_name": "skillrun",
      "before": null,
      "after": {
        "command": "skillrun",
        "args": ["router", "serve", "--mcp"]
      }
    }
  ],
  "warnings": []
}
```

约束：

- apply 前必须能 preview diff。
- apply 前必须备份原配置。
- apply 后必须能 rollback。
- 不直接把每个 `.skr` 或 capsule path 写进 MCP client 配置。

实现状态：

- v0.5.6 已实现 plan-only `skillrun consumer mount plan --client <id> --json`。
- v0.5.6 支持 `--config <path>` 作为测试和高级用户 override。
- v0.5.6 不实现 apply / rollback，也不修改真实 MCP client 配置。
- plan 目标是未来 `skillrun router serve --mcp`，因此当前输出包含 Router runtime 尚未实现的 warning。

### C005 Run History Contract

回答：

> Agent 做过什么，为什么失败，留下了什么证据？

建议 list 字段：

```json
{
  "command": "consumer runs list",
  "schema_version": "consumer.runs.list.v1",
  "registry_path": "<path>",
  "scope": {
    "kind": "registry",
    "capsule_id": null
  },
  "runs": [
    {
      "run_id": "run-...",
      "run_ref": {
        "kind": "local_run",
        "capsule_id": "refund",
        "run_id": "run-..."
      },
      "capsule_id": "refund",
      "capsule_path": "<display-path>",
      "mode": "run",
      "status": "failed",
      "ok": false,
      "error_code": "PolicyViolation",
      "started_at": "2026-05-17T10:00:00Z",
      "finished_at": "2026-05-17T10:00:01Z",
      "duration_ms": 123,
      "manifest_sha256": "<sha256>",
      "skill_sha256": "<sha256>",
      "action_sha256": "<sha256>",
      "artifact_count": 1,
      "input_included": false
    }
  ]
}
```

建议 detail 字段：

```json
{
  "schema_version": "consumer.run_detail.v1",
  "run_id": "run-...",
  "capsule_id": "refund",
  "manifest_hash": "<sha256>",
  "input_summary": {},
  "envelope": {
    "ok": false,
    "error": {
      "code": "PolicyViolation",
      "message": "approval required"
    }
  },
  "artifacts": [],
  "logs": {
    "stdout": "<path-or-null>",
    "stderr": "<path-or-null>"
  }
}
```

约束：

- list 默认不暴露完整 input。
- detail 展示 input 时必须支持摘要或脱敏策略。
- run 必须绑定 Manifest hash，避免审计时无法还原执行合同。
- UI 不应把 stdout 当作成功结果来源。

实现状态：

- v0.5.6 已实现 `skillrun consumer runs list --json [--capsule <id>] [--limit <n>]`。
- v0.5.6 不实现 `consumer runs inspect`、artifact/log content read、`--include-input` 或 global run index。

## 推荐 CLI 草案

v0.5.6 可以先采用 CLI JSON 作为 headless backend，不必立即实现 long-running daemon。

推荐命令草案：

```bash
skillrun consumer inventory --json
skillrun consumer exposure --json
skillrun consumer router status --json
skillrun consumer mount plan --client claude-desktop --json
skillrun consumer mount apply --client claude-desktop --json
skillrun consumer mount rollback --client claude-desktop --json
skillrun consumer runs list --json
skillrun consumer runs inspect <run-id> --json
```

命名判断：

- `consumer` 比 `desktop` 更好，因为这些 contract 服务 Desktop、Router、CLI automation 和后续 WebUI。
- `router` 应保留给真正的 MCP exposure runtime。
- `ui` 只应是启动或引导入口，不承载稳定数据合同。

## Daemon API 迁移路径

第一阶段：

```text
Desktop -> skillrun consumer ... --json
```

第二阶段：

```text
Desktop -> local daemon API -> SkillRun Core
```

迁移约束：

- daemon API 的 JSON shape 应尽量复用 CLI JSON shape。
- CLI JSON 是契约原型，不是临时调试输出。
- Desktop 不因为 daemon 出现而绕过 Consumer Mode。

## 与现有命令的关系

不要破坏现有黄金路径：

```bash
skillrun init
skillrun manifest
skillrun inspect
skillrun check
skillrun test
skillrun run
skillrun serve --mcp
skillrun pack
```

`consumer` 命令组只服务控制面，不替代作者路径和单 capsule 运行路径。

## 成功标准

v0.5.6 完成时，应满足：

- Desktop 不需要读取 `.skillrun/` 内部目录即可展示 capsule list、detail、readiness 和 run history。
- Router / mount / switchboard 的职责边界清楚。
- `.skr` 被定义为 import artifact，而不是 MCP runtime entry。
- registry inventory、switchboard enable state 和 readiness gate 三者的关系清楚。
- run evidence 有稳定 list/detail JSON contract。
- 所有新增 JSON surface 都有 fixture 或 contract test 计划。

1. 先 review 本文档，确认 v0.5.6 是否只做 release polish + headless contract。
2. 若通过，再把 `consumer` 命令组拆成小任务：inventory、exposure、mount plan、run history。
3. Desktop 新仓库只在这些 contract 稳定后启动 UI 原型。
