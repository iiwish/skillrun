# SkillRun v0.5.0: Language-agnostic Adapter Protocol

**文档状态**：Confirmed planning contract
**版本**：v0.5.0 design draft
**最后更新**：2026-05-14

---

## 一句话判断

v0.5.0 的核心问题不是“再支持一种语言”，而是把 **支持语言** 这件事从 Rust Core 的实现细节提升为稳定的 **Adapter Protocol**。

换句话说：

```text
v0.3 proved multiple adapters can exist.
v0.4 proved distributed capsules can be checked without importing source.
v0.5 should define how any language integrates without changing Core semantics.
```

## 为什么现在需要 v0.5.0

SkillRun 已经有 Python stable path 和 JS Action Alpha。继续用“每多一种语言就在 Core 里补一组特殊逻辑”的方式，会带来三个问题：

- Core 被语言生态细节污染，例如 package manager、schema library、module system、runtime discovery。
- 社区想支持 Ruby、PHP、Go 等语言时，不知道应该实现什么最小合同。
- Consumer Mode 的安全边界会变模糊，因为 metadata、runtime、dependency check 和 SDK helper 没有统一协议层。

v0.5.0 应该解决这个结构性问题：

> **Core 永久只认 Manifest、Adapter Protocol、IPC、Envelope 和 Artifact；语言生态通过 adapter 接入，不反向定义 SkillRun 产品语义。**

## 核心定位

v0.5.0 的交付目标应是：

> **Define and prove a language-agnostic Adapter Protocol for SkillRun capsules, with Level 0 command adapter as the first protocol-native slice.**

中文：

> **定义并验证 SkillRun 的语言无关 Adapter Protocol，并以 Level 0 command adapter 作为第一条协议原生纵向切片。**

## 分层模型

```text
SkillRun Core
  - Rust runtime
  - reads Manifest
  - creates IPC files/directories
  - validates output/error/artifact contracts
  - exposes MCP from Manifest

Adapter Protocol
  - metadata phase contract
  - run phase contract
  - runtime requirement reporting
  - error and artifact envelope requirements
  - conformance fixtures

Language Adapter
  - Python adapter
  - Node adapter
  - future Ruby/PHP/Go adapters

Language SDK
  - authoring convenience
  - typed input/output helpers
  - preflight/action wrappers
  - artifact helpers
```

关键原则：

- Core 不理解 Pydantic、Zod、Bundler、Composer、Go modules 或 npm。
- Adapter 可以理解语言生态，但必须把结果编译回 Manifest 和标准 envelope。
- SDK 是作者体验，不是 runtime authority。
- MCP 是北向 invocation surface；Adapter Protocol 是南向 execution boundary。

## v0.5.0 应该解决的问题

### P1: Adapter Protocol Contract

定义 adapter 必须支持的最小协议：

- metadata phase 输入和输出。
- run phase 输入和输出。
- IPC 环境变量。
- schema reporting。
- runtime requirement reporting。
- structured error mapping。
- artifact declaration。
- stdout/stderr discipline。
- timeout 和 exit behavior。

### P2: Adapter Capability Levels

不要要求社区一开始实现完整官方 adapter。建议定义三个层级：

```text
Level 0: command adapter
  - explicit command from config/Manifest
  - schema comes from config or Manifest
  - no dynamic metadata extraction
  - proves language agnosticism without blessing a new language ecosystem

Level 1: community adapter
  - implements metadata/run protocol
  - passes conformance fixtures
  - may have limited SDK ergonomics

Level 2: blessed adapter
  - officially maintained
  - has readiness checks
  - has docs, templates and release tests
```

这样社区可以低成本接入，同时 SkillRun 仍能保护官方承诺边界。

### P3: Conformance Test Suite

v0.5.0 的护城河不只是文档，而是 conformance：

- metadata returns input/output schema。
- run writes success envelope。
- preflight rejection maps to `PolicyViolation`。
- invalid input maps to `ValidationError`。
- malformed output maps to `ProtocolViolation`。
- stdout does not become result。
- artifacts stay inside artifact dir。
- missing runtime dependency maps to `DependencyError`。

未来任何 adapter 都应该先跑这些 fixtures。

### P4: Manifest Runtime Requirements

v0.4 已经有 runtime requirement 的雏形。v0.5 应该把它纳入 Adapter Protocol：

```yaml
runtime:
  adapter: "python"
  entrypoint: "action.py"
  protocol_version: "adapter.v1"
  requirements:
    executable:
      name: "python"
      version: ">=3.10"
    packages:
      - name: "pydantic"
        version: ">=2,<3"
        required_for: ["metadata", "runtime"]
```

注意：requirements 是诊断合同，不是 dependency install plan。

### P5: Level 0 Command Adapter

v0.5.0 应该实际交付 Level 0 command adapter。否则 v0.5 只会停留在“协议愿景”，无法证明 Core 已经从语言特判走向语言无关。

Level 0 的边界必须窄：

- 作者显式声明 command。
- input/output schema 来自 config 或 Manifest，不从源码动态提取。
- Consumer Mode 不 import action source。
- Core 仍然创建标准 IPC 文件和 artifact dir。
- command 进程仍然必须写 `SKILLRUN_OUTPUT_JSON`。
- stdout/stderr 仍然只是日志。
- dependency readiness 只诊断命令是否可执行，不安装依赖。

推荐作者形态可以是：

```json
{
  "runtime": {
    "adapter": "command",
    "command": ["ruby", "action.rb"],
    "timeout": "30s"
  },
  "input_schema": {
    "type": "object",
    "required": ["name"],
    "properties": {
      "name": { "type": "string" }
    }
  },
  "output_schema": {
    "type": "object",
    "required": ["message"],
    "properties": {
      "message": { "type": "string" }
    }
  }
}
```

这不是 Ruby support，也不是 PHP support。它是“任何进程只要遵守 SkillRun IPC/envelope contract，就可以作为低层 adapter 被 Core 调用”的最小证明。

## v0.5.0 不应该做什么

- 不做完整 TypeScript runtime。
- 不接管 npm、pip、Bundler、Composer、Go modules install flow。
- 不 vendor dependencies。
- 不做 registry。
- 不做 sandbox。
- 不新增 MCP transport。
- 不把 `.skr` 变成 runtime image。
- 不把 SDK 设计成 `serve` 框架。

v0.5.0 的核心是协议边界，不是语言生态大扩张。

## 推荐交付切片

### Slice 1: Protocol Spec

- `docs/adapter-protocol.md`
- Adapter phase definitions。
- Envelope and artifact requirements。
- Adapter capability levels。

### Slice 2: Core Boundary Refactor

- 把现有 Python/Node adapter 行为映射到同一套 protocol traits 或 internal contract。
- 不改变用户可见行为。

### Slice 3: Conformance Fixtures

- 把 Python stable 和 JS alpha 都跑过同一组 adapter conformance tests。
- conformance tests 先覆盖行为，不要求新增语言。

### Slice 4: Level 0 Command Adapter Implementation

- 实现 Level 0 command adapter。
- 允许显式 command + static schema 的最小语言无关路径。
- Consumer Mode 仍只读 Manifest，不猜测源码。
- 用至少一个非 Python/Node 示例证明能力，但不将该语言标为 blessed adapter。

## Open Questions

1. Adapter Protocol 是否需要独立 `protocol_version`，例如 `adapter.v1`？
2. metadata phase 的 transport 是否继续保持实现细节，还是公开为 adapter author API？
3. 社区 adapter 的 conformance runner 应该是 Rust integration test、CLI command，还是独立 fixture package？
4. `skillrun.config.json` 的 static schema 应该使用完整 JSON Schema，还是 v0.5 提供一个轻量 schema sugar？
5. `runtime.command` 应该只允许 argv array，还是允许 shell string？当前建议只允许 argv array。

## 成功标准

v0.5.0 完成时，读者应该能回答：

- Core 和 Adapter 的边界在哪里？
- 新语言最小要实现什么？
- 哪些能力属于 official blessed adapter？
- Consumer Mode 如何避免 metadata import 未信任源码？
- 为什么 v0.5 不是“支持一堆语言”的版本，而是“定义语言接入协议”的版本？
