# v0.5.5 Release Gate Review

## 一句话判断

v0.5.5 当前可以进入合并前最终验收：核心 Consumer Mode 出口已经统一到 Manifest 静态合同，执行、MCP 暴露和 `.skr` 分发不再接受坏 Manifest。

## Gate 标准

本轮 release gate 只验证一个问题：

> 每个 consumer-facing command 要么消费已经验证过的 Manifest，要么在执行、对外暴露或分发之前拒绝。

这不是 sandbox 审计，也不是权限系统扩展。v0.5.5 的边界仍然是：

- Author Mode 可以动态提取 metadata。
- Consumer Mode 只信任静态 Manifest。
- 声明权限用于审查和注入边界，不等同 sandbox。
- adapter timeout 是 best-effort process lifecycle 控制，不等同安全隔离。

## 关键证据

代码边界：

- `consumer::validate` 现在集中校验 source path/hash、`runtime.adapter`、`runtime.entrypoint`、`schemas.input`、`schemas.output`。
- `runtime::load_manifest` 通过 `consumer::validate` 进入 `test`、`run` 和 MCP `tools/call`。
- `serve --mcp` / `serve --mcp --dry-run` 在 CLI 层先调用 `consumer::validate`。
- `pack::create` 在创建 dist archive 前调用 `consumer::validate`。
- `readiness::evaluate` 不执行 action，不 import action source，只读取 Manifest、文件、hash 和依赖探测结果。

测试证据：

- `runtime_rejects_missing_runtime_contract_before_creating_run_records`
- `runtime_rejects_missing_runtime_entrypoint_before_creating_run_records`
- `runtime_rejects_invalid_schema_contract_before_creating_run_records`
- `check_reports_invalid_schema_contract_without_creating_run_records`
- `mcp_dry_run_fails_closed_when_manifest_schema_contract_is_invalid`
- `pack_refuses_invalid_schema_contract_before_archive_creation`
- `command_adapter_timeout_terminates_spawned_child_process`

完整验证命令：

```powershell
cargo fmt --check
cargo test
cargo clippy --all-targets -- -D warnings
git diff --check
```

## 风险与保留项

### R001 MCP stdio 启动后 Manifest 变化

`serve --mcp` 在启动前验证 Manifest，长连接运行期间不重新加载 Manifest。当前判断为可接受：server 生命周期绑定启动时 Manifest snapshot，`tools/call` 使用同一个启动时 Manifest，并不会动态信任被修改的源码。

后续 Desktop / daemon 模式如果引入热加载，必须新增 manifest reload contract。

### R002 Registry add 不强制 runnable

`registry add` 可以登记一个本地路径；这不代表 enable 或可运行。真正激活发生在 `switchboard enable`，并由 readiness gate 拒绝 invalid/stale/dependency-error。

当前判断为可接受，因为 registry 是 inventory，不是 execution permission。

### R003 Schema validator 仍是 JSON Schema 子集

Core validator 不是完整 JSON Schema 实现。v0.5.5 只承诺 SkillRun 当前生成和声明的子集，包括 `type`、`enum`、`required`、`properties`、`items`、`additionalProperties`、`minLength`、`minimum`。

当前判断为可接受，但 README / SSOT 不能暗示完整 JSON Schema 支持。

### R004 Process lifecycle 不是 sandbox

adapter timeout 已做进程树 best-effort 终止，但不能承诺完整隔离、权限 enforcement 或恶意代码防护。

当前判断为可接受，叙事必须继续保持“无护栏，不执行”的诚实边界。

## Release Gate 结论

v0.5.5 的核心合并条件基本满足。

建议下一步：

1. 跑最后一次 release validation。
2. 若无失败，将 `codex/v0.5.5-core-contract-hardening` 合并到 `main`。
3. 合并后再准备 v0.5.5 release notes，不再继续扩大 v0.5.5 范围。
