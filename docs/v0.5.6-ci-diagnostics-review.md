# v0.5.6 CI Diagnostics Review

## 一句话判断

当前 CI 诊断增强可以作为 v0.5.6 的 release polish 基线保留：它不改变成功路径，只在 `cargo test` 失败时把末尾日志压缩成 GitHub annotation，解决 v0.5.5 发布时未认证日志不可见的问题。

## 审核对象

文件：

- `.github/workflows/ci.yml`

相关步骤：

```yaml
- name: Run cargo test
  shell: bash
  run: |
    set +e
    cargo test 2>&1 | tee cargo-test.log
    status=${PIPESTATUS[0]}
    if [ "$status" -ne 0 ]; then
      tail -n 50 cargo-test.log | sed -E 's/\x1B\[[0-9;]*[mK]//g; s/%/%25/g; s/\r/%0D/g; s/$/%0A/' | {
        message=""
        while IFS= read -r line; do
          message="${message}${line}"
        done
        echo "::error title=cargo test failed::${message}"
      }
      exit "$status"
    fi
```

## 审核结论

### C001 成功路径不应被污染

当前脚本只在 `cargo test` 返回非 0 时生成 annotation。成功时不会创建 GitHub error annotation，也不会改变 job 结果。

判断：通过。

### C002 失败路径保留真实 exit code

脚本使用 `status=${PIPESTATUS[0]}` 捕获 `cargo test` 的真实退出码，而不是 `tee` 的退出码。失败后 `exit "$status"`。

判断：通过。

### C003 失败信息适合公开 CI 页面

失败时只输出 `cargo-test.log` 末尾 50 行，并做三类处理：

- 清理 ANSI 控制符。
- 转义 `%`。
- 将换行编码为 GitHub annotation 可读格式。

判断：通过。

### C004 不引入过早 CI matrix

v0.5.5 暴露的是 Linux CI 与 Windows 本地差异，但当前项目还没有足够证据需要维护完整 OS matrix。继续保留单一 `ubuntu-latest` CI 更克制。

判断：通过。

### C005 本地复现能力受限

当前 Windows 开发机没有 `bash`，因此不能在本地完整执行 GitHub Actions 的 bash 脚本语法检查。

已执行的本地检查：

```powershell
where.exe bash
```

结果：`bash not found`。

判断：可接受，但远端 branch CI 必须作为最终证据。

## 风险

### R001 annotation 仍可能被 GitHub 截断

GitHub annotation 有长度限制。当前 `tail -n 50` 是 v0.5.5 发布中从失败调试经验得到的折中值。

当前判断：可接受。不要在 v0.5.6 引入复杂 log artifact 下载或自定义脚本，除非再次遇到不可定位失败。

### R002 sed 的 ANSI 清理依赖 GitHub runner 环境

脚本依赖 `ubuntu-latest` 上的 GNU sed 行为。当前 CI runner 与项目 CI 环境一致。

当前判断：可接受。远端 CI 是最终验证来源。

## 建议

v0.5.6 不需要继续改 CI workflow。下一步应该：

1. 推送 `codex/v0.5.6-release-polish`。
2. 等待远端 `fmt`、`clippy`、`test` 全部通过。
3. 若远端 CI 通过，将 T002 视为完成。
4. 若远端 CI 失败，先读取 annotation，再决定是否修 workflow 或项目代码。
