# SkillRun v0.4: Portable Consumer Checks

**版本**：v0.4.0 milestone contract  
**状态**：Ready_For_User_Review  
**来源**：`docs/ssot.md`、v0.3.0 本地 release handoff、用户关于 dependency-aware Consumer Mode 的设计建议  
**最后更新**：2026-05-13  
**Review**：等待用户确认后才能进入任务拆分和实现。

---

## 1. 一句话判断

v0.4 验证 SkillRun 的第二个核心假设：

> **Skill Capsule 可以被分发、检查和解释，即使当前机器不能运行它。**

v0.3 证明 `Core + Manifest + IPC` 不是 Python 私有实现，而是可扩展 adapter 架构。v0.4 不继续扩语言，不做 HTTP，不做安装器；v0.4 要把 `.skr` 从“source + Manifest archive”推进到“可被 Consumer Mode 静态诊断的 skill package”。

---

## 2. 版本定位

推荐版本叙事：

| Version | Focus |
| --- | --- |
| `v0.2` | Manifest-driven MCP Skill Capsule RC |
| `v0.3` | JS Action Alpha，证明 adapter extensibility |
| `v0.4` | Portable Consumer Checks，证明 `.skr` 分发后可被安全 inspect 和 dependency-check |

v0.4 的关键词是：

- `Portable Consumer Checks`
- `Dependency-aware Consumer Mode`
- `Runtime readiness from Manifest`
- `DependencyError envelope`

不建议把 v0.4 命名为 dependency cleanup。那会把它降级成报错优化。v0.4 是产品边界，不是局部修补。

---

## 3. 产品边界

v0.4 的核心产品承诺：

> **A capsule can be inspected and dependency-checked without executing untrusted source.**

这意味着：

- `inspect` 继续回答“这个 capsule 是什么、Manifest 合同是什么”。
- `check` 成为严肃的 capsule readiness command，回答“这台机器能否消费或运行它，不能运行时缺什么”。
- `doctor` 保留为人类友好的诊断入口，可复用 `check` 的核心判断，但不替代 `check` 的语义。
- `run` 和 MCP `tools/call` 遇到 adapter/runtime dependency failure 时返回 structured `DependencyError`。
- MCP server 在单次 tool call 依赖缺失时必须保持存活。

---

## 4. 命令语义

### 4.1 `inspect`

`inspect` 只展示 Manifest contract 和 instruction-only 状态。

它不负责判断当前机器是否装了 Python、Node、Pydantic 或其他 runtime dependency。它必须继续保持 Consumer Mode 只读，不 import action source。

### 4.2 `check`

`check` 是 v0.4 的核心命令。

推荐命令：

```bash
skillrun check --cwd <capsule>
```

职责：

- 校验 capsule 文件形态。
- 校验 Manifest 是否存在且 fresh。
- 从 Manifest 读取 adapter、entrypoint、runtime requirements。
- 静态确认 source hash、example presence、entrypoint presence。
- 检查当前 host 是否具备 adapter runtime，例如 Python 或 Node。
- 检查 adapter 必需 dependency，例如 Python capsule 的 Pydantic v2。
- 输出明确 status、reason、detected value、required value 和 next step。

约束：

- 不动态 import `action.py` 或 `action.mjs`。
- 不执行 `preflight` 或 `run`。
- 不自动安装 Python、Node、Pydantic 或 npm package。
- 不从 `examples/` 反推 schema。

### 4.3 `doctor`

`doctor` 保留为人类友好诊断命令。v0.4 后建议：

- `doctor` 可以调用与 `check` 相同的 readiness engine。
- `doctor` 可以增加解释性文案和 recovery guidance。
- 自动化、CI 和 release gate 优先使用 `check`。

## 5. Manifest Dependency Contract

v0.4 需要给 Manifest 增加最小 runtime dependency contract。它不是 dependency vendoring，也不是 lockfile。

建议字段：

```yaml
runtime:
  adapter: python
  entrypoint: action.py
  timeout: 30s
  requirements:
    executable:
      name: python
      version: ">=3.10"
    packages:
      - name: pydantic
        version: ">=2,<3"
        required_for: ["metadata", "runtime"]
```

JS Alpha capsule 的最小形态：

```yaml
runtime:
  adapter: node
  entrypoint: action.mjs
  timeout: 30s
  requirements:
    executable:
      name: node
      version: ">=18"
    packages: []
```

原则：

- requirements 由 adapter defaults 生成，未来再允许 config override。
- requirements 用于诊断，不用于自动安装。
- requirements 必须进入 `.skr`，否则 `.skr` 解包后无法解释当前 host 是否 ready。
- Manifest 缺少 requirements 时，v0.4 可以降级使用 adapter defaults，但 `check` 应提示这是 legacy Manifest。

## 6. Structured DependencyError

v0.4 必须把 dependency failure 从泛化 `RuntimeError` 中分离出来。

新增或正式启用错误码：

```text
DependencyError
```

示例：

```json
{
  "ok": false,
  "error": {
    "code": "DependencyError",
    "message": "Python adapter requires Python >=3.10 with pydantic>=2,<3, but no Python executable was found.",
    "recoverable": false,
    "llm_hint": "Ask the user to install or configure the Python runtime before retrying."
  },
  "display": {
    "markdown": "Python runtime is missing. Install Python >=3.10 and pydantic>=2,<3 before running this capsule."
  }
}
```

要求：

- `run` 缺 Python、Node 或 adapter package 时返回 `DependencyError` envelope。
- `test` 同样返回 `DependencyError` envelope。
- MCP `tools/call` 将 dependency failure 映射成 MCP tool error result，但 MCP server 进程继续运行。
- `DependencyError` 不能掩盖 stale Manifest。Manifest stale 仍然先 fail closed。

## 7. Functional Requirements

### FR-001：`check` 静态诊断 Capsule

`skillrun check --cwd <capsule>` 必须在不 import action source 的前提下读取 Manifest、source hashes、entrypoint、examples 和 runtime requirements。

验收：

- fresh capsule 返回 success。
- stale Manifest 返回 failure，并指出 stale source。
- instruction-only Skill 返回 failure，并说明不是 runnable SkillRun Capsule。
- 解包 `.skr` 后可以运行 `inspect` 和 `check`。

### FR-002：Runtime Discovery

SkillRun 必须为 adapter 提供统一 runtime discovery。

验收：

- Python adapter 能报告 Python executable 是否存在、detected version、required version。
- Node adapter 能报告 Node executable 是否存在、detected version、required version。
- 缺失 executable 时，错误信息不能暴露为 `program not found` 级内部错误。

### FR-003：Adapter Package Check

SkillRun 必须能检查 adapter 必需 package。

验收：

- Python capsule 缺 Pydantic 时，`check` 失败并说明 Pydantic v2 requirement。
- Python capsule 只有 Pydantic v1 时，`check` 失败并说明 detected major version 不满足。
- JS Alpha v0.4 不要求 npm install flow，不读取 `package.json` 作为 dependency source。

### FR-004：DependencyError Runtime Envelope

`test`、`run` 和 MCP `tools/call` 依赖缺失时必须返回 structured `DependencyError`。

验收：

- CLI stdout 返回合法 error envelope。
- display markdown 不包含 stack trace。
- stderr 或 run logs 可保留底层诊断。
- MCP server 在一次 dependency failure 后仍可响应后续 request。

### FR-005：Command Boundary

`inspect`、`check`、`doctor` 三者必须边界清晰。

验收：

- README 和 docs 明确：`inspect` 看合同，`check` 看当前 host readiness，`doctor` 是人类友好诊断入口。
- `manifest` 仍属于 Author Mode，可以执行 metadata phase。
- `check` 和 `doctor` 属于 Consumer Mode，不 import source。

### FR-006：`.skr` Portable Diagnosis

`.skr` 解包后，即使缺少 runtime dependency，也必须能被解释。

验收：

- `.skr` 不 vendor dependencies。
- 解包后 `inspect` 成功展示 Manifest contract。
- 解包后 `check` 能说明当前 host 缺什么。
- 缺 dependency 不影响 source hash freshness 判断。

---

## 8. Non-Goals

v0.4 不做：

- HTTP transport。
- 自定义 REST API。
- marketplace / registry。
- signed package。
- dependency vendoring。
- Python、Node、Pydantic 或 npm package 自动安装。
- virtualenv、uv、pip、poetry、npm、pnpm、yarn 管理。
- full TypeScript runtime support。
- OS sandbox。
- 多 action 编排。

这些能力不是永远不做，但不属于 v0.4 的验证面。

## 9. Test Matrix

v0.4 测试重点不是 happy path，而是 hostile host environment。

| ID | 场景 | 必须断言 |
| --- | --- | --- |
| C001 | fresh Python capsule | `check` success，报告 adapter、entrypoint、Python detected、Pydantic satisfied |
| C002 | fresh JS capsule | `check` success，报告 adapter、entrypoint、Node detected |
| C003 | missing Manifest | `check` failure，提示运行 `skillrun manifest`，不 import action |
| C004 | stale action source | `check` failure，指出 stale source，不创建 run record |
| C005 | unpacked `.skr` | `inspect` 和 `check` 可运行，不依赖 `.git` |
| C006 | no Python executable | `check` failure；`run` 返回 `DependencyError`，不暴露 `program not found` |
| C007 | unsupported Python version | 报告 required 和 detected version |
| C008 | missing Pydantic | 报告 `pydantic>=2,<3` requirement 和修复建议 |
| C009 | Pydantic v1 only | 报告 detected package version 不满足 |
| C010 | no Node executable | JS capsule `check` failure；`run` 返回 `DependencyError` |
| C011 | path with spaces or non-ASCII | `check` 和 `run` 错误路径展示稳定 |
| C012 | MCP dependency failure | `tools/call` 返回 tool error，server 继续存活 |
| C013 | Consumer no-import guard | `check` 和 `doctor` 不 import `action.py` / `action.mjs` |

---

## 10. Success Criteria

v0.4 完成必须同时满足：

- 用户拿到别人给的 `.skr` 或 capsule 文件夹后，可以先 `inspect` 和 `check`，即使当前机器不能 run。
- 缺 Python、Node 或 Pydantic 不再表现为内部 spawn/import 错误。
- `DependencyError` 成为 CLI 和 MCP tool call 的稳定错误类型。
- MCP server 不因单个 tool call 的 dependency failure 崩溃。
- 文档能清楚说明：SkillRun 诊断依赖，但不安装依赖；`.skr` 可诊断，但不是 runtime image。

---

## 11. 后续顺序

v0.4 之后才适合讨论 HTTP：

1. `v0.4`：Portable Consumer Checks。
2. `v0.5` 或更晚：MCP Streamable HTTP transport。
3. 更晚：signed package、registry、dependency vendoring 或 reproducible runtime image。

HTTP 是放大器，不是地基。只有当本地 Consumer Mode 能稳定解释 runtime readiness 后，HTTP 才不会把本地依赖问题放大成远程调试问题。
