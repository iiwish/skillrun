# v0.5.6 Run History Contract Review

## 一句话判断

T005 不应在没有合同收敛前直接实现；当前 SkillRun 已经有可审计 run record，但它仍是 capsule-local evidence layout，不应该被 Desktop 或 Router 直接当成稳定 API 读取。

## 关键洞察

### 1. Run record 已经是证据，不是查询面

当前 runtime 会为每次 `test` / `run` / MCP tool call 创建：

```text
<capsule>/.skillrun/runs/<run-id>/
  input.json
  context.json
  output.json
  stdout.log
  stderr.log
  record.json
  artifacts/
```

`record.json` 已经包含 `run_id`、`mode`、`status`、`started_at`、`finished_at`、`duration_ms`、`capsule_dir`、`manifest_path`、`manifest_sha256`、`skill_sha256`、`action_sha256`、`permissions` 以及相对 evidence 文件名。

这足以支撑审计，但不足以直接支撑 Desktop：

- 它分散在每个 capsule 内部目录。
- 它没有全局索引。
- 它没有稳定 capsule id。
- 它把 `input.json` 作为 evidence 保存，但 list view 不应默认暴露完整 input。
- 它把 `stdout.log` / `stderr.log` 保存为日志，但 stdout 不能被解释成成功结果。

### 2. T005 的真正问题是索引边界，不是 JSON 序列化

最容易犯的错是做一个 `skillrun consumer runs list --json`，然后递归扫描 `.skillrun/runs` 并把目录路径原样输出。

这会把三个内部事实过早变成 API：

- run evidence 必须永远位于 capsule 目录下。
- run id 在所有 capsule 中全局唯一。
- Desktop 可以把文件路径当成可长期保存的引用。

更强的合同应该把这些降级成 Core 解释后的结果：

- `capsule_id` 来自 registry，而不是从路径猜。
- `run_ref` 是 opaque reference，而不是承诺目录结构。
- detail command 由 Core 读取 evidence 文件，再输出结构化 JSON。

### 3. list 和 detail 必须有不同隐私等级

`runs list` 是概览，默认不应该包含：

- 完整 input。
- 完整 output envelope。
- stdout / stderr 内容。
- artifact 文件内容。

`runs inspect` 可以返回 envelope、artifact metadata 和 log metadata，但仍应避免默认内嵌完整 input。输入只应以 `input_summary`、`input_redacted` 或后续显式 `--include-input` 形式出现。

这不是“企业安全”姿态，而是基础 DX：UI 第一屏不应该把用户传给 action 的全部业务参数扫出来。

### 4. 当前可以实现 registry-scoped list，但不应声称全局 run history

在没有 daemon / global index 前，v0.5.6 如果实现 T005，只能承诺：

> 查询当前 registry 中已登记 capsule 的本地 run evidence。

这意味着：

- 未登记 capsule 的历史 run 不进入 `consumer runs list`。
- registry entry 缺失或 Manifest 损坏时，该 capsule 的 run history 可以显示为 degraded source，但不应让整个 list 失败。
- 同一个 `run_id` 如果在多个 capsule 下出现，`inspect <run-id>` 必须返回 ambiguous，并要求 `--capsule <id>`。

## 风险 / 反例 / 误区

### R001 把 `.skillrun/runs` 布局承诺成 API

如果 JSON 直接输出 `run_dir`、`record`、`output`、`stdout`、`stderr` 并让 Desktop 自己读文件，Desktop 就会绕过 Core contract。

允许给用户展示 display path，但稳定引用应是：

```json
{
  "run_ref": {
    "kind": "local_run",
    "capsule_id": "refund",
    "run_id": "run-..."
  }
}
```

### R002 把 stdout/stderr 当成结果面

SkillRun 的结果事实是 envelope。stdout / stderr 只能是 logs。

因此 list 中最多出现：

```json
{
  "logs": {
    "stdout_available": true,
    "stderr_available": true
  }
}
```

不要在 list 中放 stdout / stderr content。

### R003 把 list 做成审计导出

Desktop 的 Envelope Explorer 第一屏需要的是 summary，不是完整审计归档。

完整导出可以是未来能力；v0.5.6 的合同不应急于解决 retention、redaction、PII、artifact content streaming、log truncation 和跨机器迁移。

### R004 不区分 missing evidence 和 failed run

如果 `record.json` 缺失或损坏，UI 应展示 degraded run source，而不是把它误判为 action failed。

建议 status：

- `succeeded`
- `failed`
- `missing-record`
- `invalid-record`
- `missing-envelope`
- `invalid-envelope`

## 更强版本

### 推荐命令草案

```bash
skillrun consumer runs list --json [--capsule <id>] [--limit <n>]
skillrun consumer runs inspect <run-id> --json [--capsule <id>]
```

命名判断：

- `runs` 放在 `consumer` 下是对的：这是 Desktop、Router automation 和 CLI automation 的 control-plane surface。
- 不要叫 `history`：history 容易暗示全局、长期、可检索数据库。
- `inspect` 比 `detail` 更符合现有 CLI 语义。

### list 合同草案

```json
{
  "command": "consumer runs list",
  "schema_version": "consumer.runs.list.v1",
  "registry_path": "<path>",
  "scope": {
    "kind": "registry",
    "capsule_id": null
  },
  "runs": [
    {
      "run_id": "run-...",
      "run_ref": {
        "kind": "local_run",
        "capsule_id": "refund",
        "run_id": "run-..."
      },
      "capsule_id": "refund",
      "capsule_path": "<display-path>",
      "mode": "run",
      "status": "failed",
      "ok": false,
      "error_code": "PolicyViolation",
      "started_at": "2026-05-17T10:00:00Z",
      "finished_at": "2026-05-17T10:00:01Z",
      "duration_ms": 123,
      "manifest_sha256": "<sha256>",
      "skill_sha256": "<sha256>",
      "action_sha256": "<sha256>",
      "artifact_count": 1,
      "input_included": false
    }
  ]
}
```

list 约束：

- 默认只扫描 registry 中的 capsule。
- 默认不返回 input、完整 envelope、log content 或 artifact content。
- `ok` 从 `output.json` envelope 读取；读不到时为 `null`，并用 degraded status 解释原因。
- `error_code` 只来自 `envelope.error.code`。
- 排序按 `started_at` 倒序；无法解析时间的记录排后。

### inspect 合同草案

```json
{
  "command": "consumer runs inspect",
  "schema_version": "consumer.runs.inspect.v1",
  "run_ref": {
    "kind": "local_run",
    "capsule_id": "refund",
    "run_id": "run-..."
  },
  "record": {
    "run_id": "run-...",
    "mode": "run",
    "status": "failed",
    "started_at": "2026-05-17T10:00:00Z",
    "finished_at": "2026-05-17T10:00:01Z",
    "duration_ms": 123,
    "manifest_sha256": "<sha256>",
    "skill_sha256": "<sha256>",
    "action_sha256": "<sha256>"
  },
  "input": {
    "included": false,
    "summary": null
  },
  "envelope": {
    "ok": false,
    "error": {
      "code": "PolicyViolation",
      "message": "approval required"
    }
  },
  "artifacts": [
    {
      "name": "receipt",
      "kind": "markdown",
      "path": "receipt.md",
      "available": true
    }
  ],
  "logs": {
    "stdout_available": true,
    "stderr_available": true
  }
}
```

inspect 约束：

- `envelope` 是事实源，stdout / stderr 只是 logs。
- artifact path 必须保持相对 artifact dir，不输出可被误认为稳定 API 的绝对内部路径。
- 如果 `run_id` 在多个 registered capsules 下匹配，返回错误：

```json
{
  "ok": false,
  "error": {
    "code": "AmbiguousRunId",
    "message": "run_id matched multiple registered capsules; pass --capsule <id>"
  }
}
```

## v0.5.6 是否应该实现

判断：可以实现 `runs list` 的最小版本，但不建议在 v0.5.6 同时实现 `runs inspect`。

理由：

- `runs list` 是 Envelope Explorer 第一屏的必要输入，范围可以控制在 registry-scoped summary。
- `runs inspect` 会立刻遇到 input redaction、artifact opening、log truncation 和 ambiguous run id 处理，容易把 v0.5.6 拉成半个 Desktop backend。
- 当前 v0.5.6 已经新增 `consumer inventory` 与 `consumer exposure`，release polish 的边界已经足够有价值。

如果继续实现，推荐只做：

```bash
skillrun consumer runs list --json [--capsule <id>] [--limit <n>]
```

并明确不做：

- `runs inspect`
- artifact content read
- log content read
- `--include-input`
- global run index
- retention / cleanup

## 实现状态

v0.5.6 已实现：

```bash
skillrun consumer runs list --json [--capsule <id>] [--limit <n>]
```

实现边界：

- 只扫描当前 registry 中已登记 capsule 的 local run evidence。
- 默认不返回完整 input、完整 envelope、stdout/stderr content 或 artifact content。
- 输出 `run_ref` 作为 opaque local reference，不要求 Desktop 读取 `.skillrun/runs`。
- 读取 `record.json` 和 `output.json` 生成 summary；缺失或损坏 evidence 降级为 `missing-record`、`invalid-record`、`missing-envelope` 或 `invalid-envelope`。
- `ok` 只来自 output envelope；`error_code` 只来自 `envelope.error.code`。
- `artifact_count` 只来自 envelope 的 `artifacts` 数组长度。
- `--capsule <id>` 按 registry id 限定 scope。
- `--limit <n>` 只截断 summary 列表，不改变 run evidence。

v0.5.6 仍不实现：

- `consumer runs inspect`
- artifact content read
- log content read
- `--include-input`
- global run index
- retention / cleanup

## 下一步动作

1. 将本文作为 T005 contract review 结果，不直接把 `.skillrun/runs` 目录写进 Desktop 合同。
2. 保持 v0.5.6 的 run history 范围为 registry-scoped `consumer runs list --json` summary。
3. 把 `consumer runs inspect` 留到 v0.5.7 或 v0.6 alpha，在 Desktop 原型需要具体 detail 时再做。
