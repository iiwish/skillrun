# v0.5.7 Public Surface Plan

## 一句话判断

v0.5.7 应该是进入 Desktop 前的公开表层收束版，不继续扩展 Core 功能；它的价值是让 README、官网、Desktop handoff 和 release 叙事对齐到同一套心智模型。

## 背景

v0.5.6 已经完成 Desktop 前置的 headless consumer contracts：

- `consumer inventory --json`
- `consumer exposure --json`
- `consumer runs list --json`
- `consumer mount plan --json`

这让 Desktop 可以作为独立项目启动，但也暴露出一个更前置的问题：

> Core 已经有 contract，但公开入口还没有达到一流开源项目应有的清晰度。

如果此时直接进入 Desktop，项目会有三个风险：

- 新用户仍把 SkillRun 理解成 FastMCP wrapper 或 YAML script runner。
- Desktop 项目容易反向污染 Core，把 UI 需求写进 runtime 边界。
- 官网、README、release notes 和 Desktop 文档对“无护栏，不执行”的解释不一致。

## 目标

v0.5.7 的目标是冻结公开表层，而不是新增 runtime capability。

它应回答：

- SkillRun 是什么，不是什么。
- 当前已实现能力和计划能力分别在哪里。
- “无护栏，不执行”里的 guardrail 到底指什么。
- Desktop 为什么是独立项目，并且只能消费 headless contract。
- v0.6 之前是否还缺一个 Router / daemon 前置版本。

## 非目标

v0.5.7 不做：

- Router runtime。
- Tauri / Desktop UI。
- `skillrun ui`。
- MCP client config apply / rollback。
- `.skr import`。
- marketplace。
- signed package trust。
- dependency installation / vendoring。
- OS sandbox。
- 新官方领域包实现。

## 当前三处复审结论

### 主仓库

状态：已完成第一轮公开入口重写。

已完成：

- English README 从历史记录式文档改成项目入口。
- 中文 README 对齐“面向 AI Agent 的 SOP-backed skill runtime”。
- 当前版本说明更新到 v0.5.7。
- README 明确 v0.5.7 不包含 Desktop、daemon、Router runtime、真实 MCP client config mutation、sandbox、marketplace 或 signed package trust。
- README 明确 Desktop 是独立项目，应消费 headless JSON surface。

需要补齐：

- 发布前将 v0.5.7 release notes 从 `Release candidate` 改为最终发布状态。
- crate / CLI version 已提升到 `0.5.7`。
- 发布前跑完整 release validation，而不是只跑文档检查。

### 官网

状态：可由用户手动提交。

已完成：

- 官网版本从 v0.5.4 更新到 v0.5.6。
- 首页 milestone 更新为 Headless Consumer Contracts。
- 中文和英文首页同步强调 v0.5.6 的消费者控制面和非 UI 边界。
- 英文首页叙事统一为 `No guardrail, no execution`，避免 `No boundaries` 造成误读。

需要补齐：

- 用户手动提交官网仓库变更。
- 官网后续应避免把 v0.6 Desktop 写成已发布能力。

### Desktop 项目

状态：已具备规划启动条件，但不应进入重 UI 开发。

已完成：

- 在 `skillrun-desktop/docs/desktop-readiness.md` 写入 Desktop readiness 草案。
- 明确 Desktop 是本地 Skill Capsule 消费控制台，不是 author IDE、marketplace 或 runtime owner。
- 明确 Desktop 只能消费 CLI JSON / future daemon API，不读取 `.skillrun/` 内部目录，不直接解析 Manifest YAML，不直接执行 action。

需要补齐：

- Desktop 初始化 git 后，把 readiness 文档提交为第一个治理文档之一。
- Desktop alpha 前确认是否需要 `router serve --mcp` 的最小可运行版本。

## v0.5.7 工作图

### T001 README Public Entry

状态：完成。

验收：

- README 30 秒内能说明为什么 SkillRun 不是 FastMCP。
- README 明确 Skill Capsule、Manifest-driven runtime、SOP-backed skill。
- README 明确 Consumer Mode 的安全叙事不是 sandbox 承诺。

### T002 Website Version Alignment

状态：等待用户手动提交。

验收：

- 官网不再停留在 v0.5.4。
- 官网不把 Desktop / Router runtime 当作已发布功能。
- 官网中英文叙事使用同一组边界词。

### T003 Desktop Handoff

状态：本地草案完成，等待 Desktop repo 初始化后提交。

验收：

- Desktop 项目知道哪些 surface 可以消费。
- Desktop 项目知道哪些内部目录和概念不能碰。
- Desktop 不把 switchboard enabled 显示成 trusted / sandboxed。

### T004 Docs Index And Release Planning

状态：完成第一轮。

验收：

- `docs/README.md` 收录 v0.5.7 计划。
- `RELEASE_NOTES.md` 包含 v0.5.7 `Release candidate` entry。
- release notes 明确这是 public surface / docs / handoff release，而不是 runtime expansion。

### T005 Pre-Desktop Gap Decision

状态：待决策。

需要回答：

- v0.5.8 是否先做最小 Router runtime。
- v0.5.8 是否先做 `consumer runs inspect`。
- v0.6 Desktop alpha 是否允许先调用 CLI JSON，而不等待 daemon API。

当前判断：

- 如果 Desktop 第一屏只做 Switchboard、Inventory、Exposure Preview、Mount Plan Preview，可以直接启动 v0.6 alpha。
- 如果 Desktop 第一屏必须做到真实一键挂载和可运行 Router，则需要 v0.5.8 Router MVP。
- `consumer runs inspect` 可以推迟到 Desktop 的 Envelope Explorer 第二阶段，不应阻塞 alpha。

## Release Gate

v0.5.7 可以合并 / 发布前必须满足：

- 主仓库 README、中文 README、docs index、release notes 一致。
- 官网版本和公开叙事已由用户手动提交。
- Desktop readiness 文档已存在，且边界与 README 一致。
- 文档链接检查通过。
- `git diff --check` 通过。
- 如果创建 `v0.5.7` tag，则 crate / CLI version、release notes 和 tag 必须一致。
- release notes 明确 Manifest IR version 与 Adapter Protocol version 是否变化；预计不变化。

## 下一步建议

1. 先复审 v0.5.7 是否需要补最小 Router MVP；当前默认不在 v0.5.7 做。
2. 再决定 v0.5.8 是 Router MVP，还是直接进入 v0.6 Desktop alpha。
3. Desktop 项目初始化 git 后，把 readiness 文档作为项目宪法级文档提交。
