# SkillRun v0.6.0: Consumer Era 愿景

**文档状态**：Planning baseline after v0.5.9
**版本**：v0.6.0 planning baseline
**最后更新**：2026-05-18

---

## 一句话判断

v0.6.0 的核心不是“给 CLI 加一个漂亮界面”，而是把 SkillRun 从已经具备 Router 和 Safe Mount Apply 的 Manifest-driven runtime，推进到消费者可理解、可挂载、可开关、可追溯的本地 skill control plane。

主题：

> **Consumer Era: 让消费者在执行前看得懂、挂得上、关得掉、追得回。**

v0.6.0 不改变 SkillRun 的核心身份。SkillRun 仍然是 Manifest-driven Skill Capsule runtime；v0.6.0 只是在 Core 之上补齐消费者入口、MCP 挂载层、本地可视化审查和官方场景样板。

## 目标用户

v0.6.0 面向 Skill 消费者，而不是优先面向 Skill 作者。

核心用户包括：

- 想使用别人分发的 Skill Capsule，但不想手写 MCP 配置的个人用户。
- 想在 Claude Desktop、Cursor 或其他 MCP client 中统一管理本地技能的 AI power user。
- 想审查 Agent 执行动作、失败原因、产物和运行证据的团队负责人。
- 想体验官方领域包的非研发用户，例如外卖商家运营、电商运营、客服主管或门店运营人员。

v0.6.0 不把普通用户包装成“安全运行任意第三方代码”的目标。运行第三方 action 仍然意味着执行第三方代码；SkillRun 提供的是 Manifest 约束、Consumer Mode 诊断、preflight、structured envelope、artifact 和 run evidence，不是 OS sandbox。

## 产品边界

v0.6.0 应避免三个方向：

- 不做通用 Agent framework。
- 不做 marketplace 或 registry 的第一版。
- 不把 Tauri/Desktop 需求反向污染 Rust Core、Manifest、Adapter Protocol 或 IPC 合同。

v0.6.0 应专注四件事：

```text
inspectable   看得懂：Manifest、permissions、dependencies、source hashes
mountable     挂得上：一键安装 MCP router profile
switchable    关得掉：启用/禁用 capsule，不暴露被关闭的 tool
auditable     追得回：run records、envelopes、artifacts、preflight results
```

## 一键挂载模型

一键挂载不应该直接把 `.skr` 挂到 Claude Desktop、Cursor 或其他 MCP client。

`.skr` 是 source + Manifest archive，是分发物，不是稳定运行入口。真正应该被 MCP client 挂载的是本地 SkillRun Router/Gateway。

推荐模型：

```text
Claude / Cursor / MCP client
        |
        v
SkillRun Local Router
        |
        v
Enabled Capsule Registry
        |
        v
Skill Capsule
  - Manifest
  - source hashes
  - runtime adapter
  - IPC/envelope/artifacts
```

角色划分：

```text
.skr                = install/import artifact
capsule registry    = local inventory and state
router              = MCP exposure surface
switchboard         = local policy/control plane
run records         = evidence and audit trail
```

用户体验可以是一键，但系统行为应是多步：

```text
Import .skr
  -> validate Manifest and source hashes
  -> register capsule
  -> check local readiness
  -> enable capsule
  -> expose through SkillRun Router profile
  -> install or update MCP client config
```

MCP client 配置中应尽量只出现一个 SkillRun Router，而不是每个 capsule 都直接改一次 client 配置。这样 v0.6 才能支持开关、升级、审查、禁用、profile、回滚和统一运行记录。

## Tauri 客户端形态

Tauri 客户端可以从第一天独立，但必须通过稳定本地接口调用 SkillRun Core。

推荐仓库边界：

```text
skillrun
  - Rust CLI/Core
  - Manifest generation
  - Consumer Mode
  - Adapter Protocol
  - runtime
  - pack
  - local router
  - headless consumer JSON contracts

skillrun-desktop
  - Tauri shell
  - Capsule Switchboard
  - MCP Mount Manager
  - Envelope Explorer
  - official pack browser
```

v0.6.0 alpha 可以先让 Desktop 调用 `skillrun ... --json`。daemon API 可以作为后续演进，但不应成为 v0.6.0 的前置条件。UI 不应该直接读取 `.skillrun/` 内部目录结构作为稳定接口。

CLI 可以保留：

```bash
skillrun ui
```

但 `skillrun ui` 只负责启动 local daemon、打开 Web Dashboard 或引导用户安装 Desktop 客户端；UI 逻辑不应嵌进 Core。

## Local Console 核心视图

v0.6.0 的 UI 不应该做作者 IDE。它应该是消费者控制台。

### Capsule Switchboard

回答：

> 当前有哪些 capsule？哪些启用？哪些禁用？哪些不能运行？

能力：

- 查看 capsule 名称、版本、来源、Manifest hash、adapter、readiness。
- 启用或禁用 capsule。
- 禁用后的 capsule 不应暴露为 MCP tool。
- 展示 declared env、declared permissions、entrypoint 和 source hash。

### MCP Mount Manager

回答：

> SkillRun Router 是否已经挂载到 Claude Desktop、Cursor 或其他 MCP client？

能力：

- 识别支持的 client profile。
- 预览配置 diff。
- 应用配置变更。
- 备份和回滚配置。
- 显示当前 router endpoint 和暴露的 enabled capsules。

### Envelope Explorer

回答：

> Agent 做过什么？被什么拦截？产出了什么证据？

能力：

- 按 capsule、时间、status、error code 过滤 run records。
- 展示 structured output/error envelope。
- 展示 `PolicyViolation`、`ValidationError`、`DependencyError`、`ProtocolViolation` 等错误分布。
- 展示 preflight 拦截次数、执行耗时、artifact 链接。
- 输入展示必须支持摘要和脱敏，避免把敏感入参无控制地暴露在 UI 中。

### Capsule Detail

回答：

> 这个 skill 到底承诺了什么？它什么时候不该执行？

能力：

- 展示 `SKILL.md` 摘要。
- 展示 Manifest-derived tool description。
- 展示 input/output schema。
- 展示 examples。
- 展示 permissions、runtime requirements 和 Consumer Mode 检查结果。

## 官方场景策略

v0.6.0 需要官方场景样板，但不应急于做 marketplace。

官方场景不应只是 `examples/`，而应被组织成：

> **Official Capsule Packs**

每个 pack 都应该回答：

- 目标用户是谁？
- 危险动作是什么？
- 为什么普通 OpenAPI/MCP wrapper 不够？
- SOP 如何进入 description、schema、preflight 和 error recovery？
- 运行后留下什么 artifact 和 evidence？
- 国内平台和海外平台分别如何映射？

### 优先领域：外卖商家多平台运营包

推荐作为 v0.6 旗舰领域包：

> **Food Delivery Merchant Ops Pack**

中文：

> **外卖商家多平台运营包**

目标用户：

- 同时经营美团、饿了么、京东到家、抖音本地生活等平台的外卖商家。
- 需要跨平台管理菜单、价格、库存、促销、订单异常和差评回复的门店运营人员。

核心判断：

> 这类场景的价值不是“把平台 API 变成 tool”，而是给跨平台经营动作加上统一 SOP、preflight、envelope 和 evidence。

候选 capsules：

- `menu_consistency_check`：跨平台菜单、价格、规格、上下架状态和图片缺失检查。
- `safe_menu_publish`：菜单发布前检查毛利底线、敏感词、图片、库存和平台差异。
- `store_status_switchboard`：爆单、后厨故障、极端天气下建议暂停营业、延长出餐或售罄 SKU。
- `order_exception_triage`：处理超时、缺货、催单、退款和骑手异常。
- `promotion_margin_guard`：检查满减、折扣和平台活动是否击穿毛利。
- `review_response_gate`：生成差评回复草稿，并在食品安全、赔偿、法律风险场景拦截。
- `daily_store_brief`：生成多平台 GMV、订单、取消、差评、超时、缺货和爆品日报。
- `food_safety_evidence_pack`：检查证照、过敏原、菜品描述和图片合规证据。

平台接入策略：

```text
Domain Skill Capsule
  -> Canonical domain schema
    -> Platform connector
      -> Meituan / Ele.me / JD Daojia / Douyin / mock / CSV
```

v0.6 不应被平台授权卡死。第一版应至少支持：

- `mock` connector：官方 demo 可跑。
- `csv/json` connector：商家可用导出数据体验。
- experimental platform connector：只在授权和公开接口可验证时提供。

## OpenAPI-assisted Capsule Generator

v0.6 可以探索 OpenAPI 辅助生成，但不能把它定义成 SkillRun 主卖点。

错误定位：

```text
OpenAPI -> trusted skill
```

正确定位：

```text
OpenAPI operation
+ SOP template
+ permission profile
+ example input
+ preflight questions
= capsule draft
```

推荐名称：

> **OpenAPI-assisted Capsule Generator**

它生成的是 capsule draft，而不是已经可信的 `.skr`。

CLI 草案：

```bash
skillrun generate openapi \
  --spec openapi.yaml \
  --operation updateMenuItem \
  --scenario food-delivery-merchant-ops \
  --output capsules/update_menu_item
```

生成后必须提示：

```text
Generated capsule draft.
Review SKILL.md, preflight, permissions and examples before running `skillrun manifest`.
```

OpenAPI 只能说明接口怎么调用，不能说明什么时候不该调用。SkillRun 的不可替代性来自 SOP、preflight、structured error、artifact 和 run evidence。

## v0.6.0 非目标

v0.6.0 不做：

- 不做中心化 marketplace。
- 不做完整 registry trust。
- 不做 signed package 强制信任体系。
- 不做 OS sandbox。
- 不做 dependency vendoring。
- 不做 runtime image。
- 不做完整 authoring IDE。
- 不做多 Agent 编排框架。
- 不把 OpenAPI-to-MCP 作为主卖点。
- 不把 UI 直接绑定到 `.skillrun/` 内部文件布局。

## v0.5.x 前置条件

v0.5.9 之后，v0.6.0 的 Core 前置条件已经从“建立控制面”转为“冻结 Desktop 可依赖的控制面合同”。

已具备：

- machine-readable `--json` 输出。
- 本地 capsule registry。
- capsule enable/disable 状态。
- 本地 SkillRun Router runtime。
- plan-first MCP client mount profile。
- Claude Desktop Safe Mount Apply / Rollback。
- run record list 能力。
- Consumer Mode trust wording 和 envelope 字段稳定化。

仍建议在进入 Desktop alpha 前补齐：

- `consumer runs inspect <run-id> --json`，支撑 Envelope Explorer 查看单次 envelope、artifact 和日志摘要。
- `.skr import` 或等价的本地 capsule import flow，避免 Desktop 直接操作 capsule 文件夹。
- Router / mount / registry JSON 合同的 public-doc refresh，避免 Desktop 消费过期叙事。
- 官方场景 pack 的 spec，而不是只存在零散 examples。

没有这些补齐项，Tauri/Desktop 仍能启动，但会被迫绕过 Core 合同，退化成漂亮外壳。

## 成功标准

v0.6.0 完成时，用户应该能做到：

- 导入一个 `.skr` 或官方 capsule pack。
- 在 UI 中看懂它的 Manifest、permissions、dependencies 和风险边界。
- 一键把 SkillRun Router 挂载到 MCP client。
- 启用或禁用某个 capsule。
- 看到 Agent 调用后的 envelope、artifact、preflight 拦截和 run evidence。
- 在不理解 Rust、MCP 或 adapter 细节的情况下，判断一个 capsule 是否值得执行。

v0.6.0 的里程碑意义不是“SkillRun 有界面了”，而是：

> **SkillRun 开始拥有消费者控制面。**
