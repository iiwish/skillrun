# SkillRun v0.5.2: Consumer JSON Surface

**文档状态**：Implemented Contract
**版本**：v0.5.2 implementation baseline
**最后更新**：2026-05-15

---

## 一句话判断

v0.5.2 的核心不是“给几个命令加 `--json`”，而是为 v0.6 Consumer Era 提供第一层稳定、只读、可自动化的机器接口。

主题：

> **Consumer JSON Surface: 让 Desktop、Router 和自动化不用解析人类终端文本。**

v0.5.2 不引入 registry、router、daemon 或 UI。它只把现有 Consumer Mode 的静态诊断和合同展示能力，暴露为稳定 JSON。

## 背景

v0.5.1 已经统一了 guardrail、trust model 和 `output` envelope 字段。v0.6.0 需要消费者控制面，但 UI 不应该读取 `.skillrun/` 内部目录，也不应该解析 `inspect`、`check`、`doctor` 的人类文本输出。

因此 v0.5.2 的工作是建立 headless contract：

```text
SkillRun CLI human text
        +
SkillRun CLI --json machine surface
        ->
future Desktop / Router / automation
```

## 范围

v0.5.2 应交付：

- `skillrun inspect --json`
- `skillrun check --json`
- `skillrun doctor --json`
- 相关 JSON contract 文档与 tests

v0.5.2 可以明确说明：

- `skillrun test` 和 `skillrun run` 当前已经输出标准 output/error envelope，本身就是 machine-readable JSON。
- v0.5.2 不额外包一层 run JSON wrapper，避免破坏现有 CLI/MCP 语义。
- run record query/index 留给后续版本，建议进入 v0.5.3 或 v0.5.4 前重新评估。

## 命令合同

### `skillrun inspect --json`

目标：

> 输出一个 capsule 的 Manifest contract 摘要，不执行 action source。

成功时输出 JSON object：

```json
{
  "command": "inspect",
  "cwd": "...",
  "status": "runnable",
  "manifest": {
    "path": ".skillrun/manifest.generated.yaml",
    "present": true,
    "freshness": "fresh"
  },
  "skill": {
    "name": "refund",
    "sop_hash": "..."
  },
  "sources": {
    "skill": {
      "path": "SKILL.md",
      "sha256": "..."
    },
    "action": {
      "path": "action.py",
      "sha256": "..."
    }
  },
  "schemas": {
    "input": "present",
    "output": "present"
  },
  "runtime": {
    "adapter": "python",
    "entrypoint": "action.py",
    "timeout": "30s"
  },
  "permissions": {
    "files": {
      "read": [],
      "write": [".skillrun/runs/**"]
    },
    "network": {
      "outbound": []
    },
    "env": {
      "read": []
    }
  },
  "examples": [
    {
      "id": "default",
      "input": "examples/default.input.json"
    }
  ],
  "preflight": "present",
  "tool": {
    "name": "refund",
    "description": "Execute the refund SkillRun capsule."
  }
}
```

状态值应与 human `inspect` 保持一致：

- `runnable`
- `invalid-runnable`
- `instruction-only`

`invalid-runnable` 必须包含 `reason`，并保持 exit code 0，与现有 human inspect 行为一致。

### `skillrun check --json`

目标：

> 输出当前 host 是否能消费或运行 capsule 的 readiness report。

JSON 应直接反映 readiness engine，而不是重新发明一套 UI 字段。建议字段：

```json
{
  "command": "check",
  "ok": true,
  "cwd": "...",
  "status": "ok",
  "manifest": {
    "path": ".skillrun/manifest.generated.yaml",
    "present": true,
    "freshness": "fresh"
  },
  "files": {
    "skill": true,
    "python_action": true,
    "node_action": false,
    "typescript": false,
    "config": true
  },
  "runtime": {
    "adapter": "python",
    "entrypoint": "action.py"
  },
  "requirements": {
    "present": true,
    "executable": {
      "name": "python",
      "version": ">=3.10"
    },
    "packages": []
  },
  "dependency_checks": [],
  "source_checks": [
    {
      "path": "SKILL.md",
      "status": "fresh"
    }
  ],
  "example_checks": [
    {
      "path": "examples/default.input.json",
      "present": true
    }
  ],
  "next_step": "none",
  "note": "readiness reads Manifest, files and hashes only; it does not run or import action source."
}
```

Exit code 必须与 human `check` 保持一致：`ok=true` 时 0，`ok=false` 时非 0。

### `skillrun doctor --json`

目标：

> 给自动化和 Desktop 提供同一份 readiness data。JSON 不需要复制 human doctor 的友好排版。

建议 `doctor --json` 与 `check --json` 共享同一个 JSON schema，只把 `command` 字段设为 `doctor`。这可以避免 Desktop 因两个诊断入口维护两套数据模型。

## 非目标

v0.5.2 不做：

- 不做 capsule registry。
- 不做 enable/disable。
- 不做 SkillRun Router。
- 不做 MCP client config 修改。
- 不做 local daemon。
- 不做 Tauri/Desktop。
- 不做 run record query/index。
- 不改变 `test` / `run` 的标准 output/error envelope。
- 不改变 Manifest schema。
- 不新增 sandbox、registry trust 或 dependency install 行为。

## 设计约束

- Human output 必须保持兼容；新增 `--json` 不应改变默认输出。
- JSON 字段应使用 snake_case，匹配 Rust/serde 习惯和现有 run record。
- JSON 输出必须是单个 object，方便 CLI 调用者解析。
- JSON 模式应适合未来 Desktop 直接消费，而不是为当前文本排版服务。
- `inspect --json` 和 `check --json` 均不得 import action source。
- 错误路径仍应使用 stderr + non-zero exit code；后续版本可再讨论 machine-readable CLI error envelope。

## Contract Fixture 规则

v0.5.4 起，Desktop-facing JSON surface 需要用 fixture 冻结。fixture 不追求保存每台机器的原始输出，而是保存归一化后的 contract：

- 绝对路径归一化为 `<capsule>` 或 `<skillrun_home>`。
- hash 归一化为 `<sha256>`。
- 本机依赖版本归一化为 `<detected>`。
- registry 时间戳归一化为 `<timestamp>`。

兼容性规则：

- 新增字段通常是 additive change，可以接受，但应同步 fixture 和文档。
- 字段删除、改名、类型变化、状态枚举变化是 breaking change，必须显式进入版本语义讨论。
- human text 不属于 Desktop contract；Desktop、Router 和自动化只能依赖 `--json`、run record 或后续 Core API。

## 验收标准

- `skillrun inspect --json --cwd path/to/capsule` 输出合法 JSON，且不 import action source。
- `skillrun inspect --json` 支持 runnable、invalid-runnable 和 instruction-only 状态。
- `skillrun check --json --cwd path/to/capsule` 输出 readiness JSON，且 exit code 与 `ok` 一致。
- `skillrun doctor --json --cwd path/to/capsule` 输出与 check 同 schema 的 readiness JSON。
- human text output 保持现有测试通过。
- 新增 tests 覆盖 JSON contract。
- `cargo test` 通过。

## 后续衔接

v0.5.2 完成后，后续版本应按这个顺序推进：

```text
v0.5.3  Capsule Registry + Switchboard
v0.5.4  SkillRun Router + Mount Profiles
v0.5.5  Official Pack Foundations
v0.6.0  Consumer Era local console
```

v0.5.2 的交付不是终点，而是让 v0.6 的消费者控制面不再依赖 brittle text parsing。
