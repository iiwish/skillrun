# SkillRun v0.5.4: Core Stabilization Audit

**文档状态**：Completed Stabilization Baseline
**版本**：v0.5.4 audit entry
**最后更新**：2026-05-15

---

## 一句话判断

v0.5.4 不应该继续加功能；它应该把整个 SkillRun 项目打磨成 Desktop、Router 和官方场景包可以依赖的稳定 Core。

这里的 Core 指整个 `skillrun` 项目：Rust CLI/runtime、Manifest、Adapter Protocol、IPC、Envelope、Consumer JSON Surface、Registry/Switchboard、MCP、`.skr` packaging、examples、docs 和 release process。

## 审核结论

SkillRun 当前已经有正确的项目骨架：边界清晰、测试量足够、Consumer Mode 安全叙事比较诚实，v0.5.3 的 registry/switchboard 也为 Desktop 做了必要铺垫。

但如果目标是“v0.5.x 内核稳定版”，现在还不能直接进入 Desktop。主要原因不是功能缺失，而是几个外部 contract 还没有硬化：

- `command` adapter 的 readiness probe 会执行任意声明 executable 的 `--version`。
- Level 0 `command` adapter 的 static schema 尚未由 Core 统一执行校验。
- Consumer JSON surface 还缺少显式 schema/fixture/golden contract。
- registry/switchboard list 对坏 Manifest 的容错边界还不够完整。
- release/version 叙事与 `Cargo.toml`/`.skr` artifact version 仍然混在一起。

## 审核范围

本轮审核覆盖：

- CLI surface：`init/manifest/inspect/check/doctor/test/run/serve/pack/registry/switchboard`
- Manifest IR：字段、版本、source hash、runtime requirements
- Adapter Protocol：Python stable、JS alpha、Level 0 command
- Runtime IPC：input/context/output/artifact 文件协议
- Envelope：success/error、structured error、MCP tool result projection
- Consumer Mode：stale Manifest、instruction-only、dependency diagnostics
- Registry/Switchboard：本地 inventory 和 exposure intent
- MCP：stdio server、tool/resource contract、dry-run output
- Packaging：`.skr` archive 边界
- Tests/docs/release：是否足够支撑 Desktop 作为独立项目消费 Core

## 关键发现

### P0: command adapter readiness 不应执行任意 executable

位置：

- `src/readiness.rs:445`
- `src/readiness.rs:446`

当前 `detect_command_executable(name)` 通过 `Command::new(name).arg("--version").output()` 检测 command adapter 的 executable。

这对 Python/Node 这类 blessed runtime 可以接受，但对 Level 0 `command` adapter 过于危险。因为 command adapter 的 executable 来自 capsule Manifest/config，消费者运行 `check`、`doctor` 或 `switchboard enable` 时，可能在“以为只是静态检查”的阶段执行任意第三方命令。

这会削弱 SkillRun 最重要的安全叙事：

```text
Consumer Mode 不为 metadata import 未信任源码
```

更精确地说，当前问题不是 import action source，而是 readiness probe 执行了未信任 command executable。对 Desktop 来说，这会让“点一下启用前检查”变成隐式执行面。

建议 v0.5.4 修复：

- Python/Node 可以保留 `--version` probe。
- command adapter 默认只做 PATH resolution / executable presence check，不运行 `--version`。
- 若未来需要运行 probe，必须是 Manifest 中显式声明的 adapter capability，且 UI/CLI 文案要说明它会执行进程。

### P0: Core 没有统一执行 JSON Schema 校验

位置：

- `src/config.rs:306`
- `src/config.rs:308`
- `src/config.rs:312`
- `src/runtime.rs:295`
- `src/runtime.rs:312`
- `src/runtime.rs:313`

当前 `input_schema` / `output_schema` 会进入 Manifest/MCP surface，Python/JS adapter 也在 SDK wrapper 中做校验。但 Core runtime 的 `adapter_envelope` 成功路径只检查：

- `ok: true`
- adapter exit status
- artifact containment

它没有统一按 Manifest `schemas.output` 校验 output，也没有在 dispatch 前按 Manifest `schemas.input` 校验 input。

这对 `command` adapter 尤其关键：Level 0 command adapter 的 schema 是 static schema，不来自 SDK。如果 Core 不执行 schema 校验，`command` adapter 就可以返回任意 `output` 并被当成成功。

这会让 “typed input/output” 退化成 MCP 描述字段，而不是 runtime contract。

建议 v0.5.4 修复：

- 引入轻量 JSON Schema validation 依赖或实现 v0 支持子集。
- `run/test/mcp tools/call` 在启动 adapter 前校验 input。
- `ok: true` 成功 envelope 在写 run record 前校验 output。
- 失败时返回 `ValidationError` 或 `ProtocolViolation`，需要明确区分：
  - 用户输入不符合 input schema：`ValidationError`
  - adapter 输出不符合 output schema：`ProtocolViolation`

### P1: registry/switchboard 对坏 Manifest 仍可能被单条 entry 拖垮

位置：

- `src/registry.rs:155`
- `src/registry.rs:161`
- `src/registry.rs:162`
- `src/registry.rs:232`
- `src/registry.rs:238`
- `src/registry.rs:239`
- `src/registry.rs:330`
- `src/registry.rs:332`
- `src/registry.rs:428`

v0.5.3 pre-merge hardening 已修复 missing path：路径丢失会作为 per-capsule `readiness.status="missing-path"` 返回。

但 `capsule_view` 仍会把 `readiness::evaluate` 或 `read_manifest` 的其他错误向上传播。典型场景：某个已注册 capsule 的 Manifest YAML 损坏，`registry list --json` 仍可能整体失败。

对 Desktop 来说，inventory list 是控制面事实源。单个坏 capsule 应该显示为 `invalid-manifest` / `unreadable-manifest`，不应该让整个页面不可渲染。

建议 v0.5.4 修复：

- `registry list --json` 和 `switchboard list --json` 对每个 capsule 独立降级。
- `registry inspect <id> --json` 也优先返回结构化 bad-entry view。
- `switchboard enable <id>` 继续 fail closed。

### P1: Consumer JSON Surface 缺少显式 contract fixture

当前已有大量行为测试，`cargo test -- --list` 显示覆盖面很强。但 `inspect --json`、`check --json`、`doctor --json`、`registry list --json`、`switchboard list --json` 还主要靠 Rust test 中的局部字段断言，而不是稳定 JSON contract fixture。

这对 Desktop 不够。Desktop 新项目一旦接入这些 JSON，字段就会变成跨仓库 contract。缺少 fixture 会导致 Core 改字段时无意破坏 UI。

建议 v0.5.4 补齐：

- `contracts/json/*.schema.json` 或 `tests/fixtures/contracts/*.json`
- 至少覆盖：
  - runnable Python capsule
  - runnable JS capsule
  - instruction-only
  - stale Manifest
  - dependency-error
  - registry empty
  - registry missing-path
  - switchboard enabled/disabled
- 建立“字段新增可以，字段改名/删除必须显式版本化”的规则。

v0.5.4 执行结果：

- 已新增 `tests/consumer_json_contracts.rs`。
- 已新增 `tests/fixtures/contracts/*.json` 初版，覆盖 `inspect/check/doctor`、instruction-only、registry enabled 和 switchboard enabled。
- fixture 会归一化路径、hash、本机依赖版本和 registry 时间戳，避免把本机环境误当成 contract。

### P1: release/version 边界还不适合稳定版

位置：

- `Cargo.toml:3`
- `src/manifest.rs:127`
- `src/manifest.rs:128`
- `src/manifest.rs:154`

当前 crate/package version 是 `0.5.0`，但文档和 release notes 已经推进到 v0.5.3/v0.5.4 语境。Manifest `manifest_version` 和 IPC `protocol_version` 仍是 `0.1.0`。

这不一定是 bug：artifact version、crate version、Manifest IR version、Adapter Protocol version 本来就应该分离。

问题是：当前文档没有把这些版本层级明确冻结。Desktop 和 `.skr` artifact 会看到 `skillrun 0.5.0`，而项目叙事在说 v0.5.3/v0.5.4，很容易误解。

建议 v0.5.4 明确：

- `Cargo.toml version` 是 binary/artifact version。
- `manifest_version` 是 IR schema version。
- `runtime.protocol_version` 是 IPC/adapter protocol version。
- docs milestone version 是 planning/release line。
- 进入稳定版前决定是否把 crate version 提升到 `0.5.4` 或继续保持 `0.5.0` release candidate。

v0.5.4 执行结果：

- 已在 `docs/release-policy.md` 明确 binary/crate version、Git tag、milestone version、Manifest IR version 和 IPC/Adapter Protocol version 的区别。
- 已在 README 和 `RELEASE_NOTES.md` 明确当前 binary/crate version 仍是 `0.5.0`，Manifest IR 与 IPC protocol version 仍是 `0.1.0`。
- Desktop 边界已冻结为独立项目消费 Core contract，而不是在 Core repo 内直接演化 UI。

### P2: MCP tool result 会把 SkillRun envelope 投影成 text

位置：

- `src/mcp.rs:369`
- `src/mcp.rs:377`
- `src/mcp.rs:394`

MCP `tools/call` 当前把 SkillRun envelope 转成 MCP text content，并通过 `isError` 表示失败。这对通用 MCP client 友好，但对 Envelope Explorer 来说还不够直接。

这不是 v0.5.4 必修 bug，因为 CLI/run record 仍保留完整 envelope。但 Desktop 如果主要从 MCP client 侧接入，会丢失结构化 envelope 的一等表达。

建议：

- v0.5.x 先不改变 MCP 输出，避免兼容风险。
- v0.5.4 文档明确：Envelope Explorer 应读取 SkillRun run records / Core JSON API，而不是从 MCP text content 反解析。
- v0.6 Desktop 不应把 MCP tool text 当作审计数据源。

## 稳定化版本切分

### v0.5.4 Core Contract Audit and Fixes

目标：修掉会破坏 Core 信任边界和 Desktop contract 的问题。

必须完成：

- command adapter readiness 不执行 arbitrary executable。
- Core 统一执行 input/output schema validation。
- registry/switchboard list 对 bad capsule entry per-entry 降级。
- Consumer JSON contract fixtures 初版。
- version semantics 文档冻结。

### v0.5.5 Core Hardening Matrix

目标：把所有核心命令纳入稳定矩阵。

建议完成：

- `cargo clippy --all-targets -- -D warnings` 纳入 release validation。
- README 中的 cargo run smoke matrix 自动化。
- `.skr` unpack 后 inspect/check/test/run/mcp dry-run contract 固化。
- Windows path、relative path、Unicode path、空 PATH、缺依赖场景补足。

### v0.5.6 Headless Desktop Backend

目标：为 Desktop 新项目提供最小稳定后端，但不做 UI。

建议完成：

- Router/daemon 或 headless service 的最小 contract。
- Switchboard `enabled` 被 Router 实际消费。
- Envelope/run record list/query JSON。
- MCP mount profile 只生成配置建议或 dry-run preview，不直接做高风险自动写入。

## Desktop 项目边界

Desktop 应新起项目，例如 `skillrun-desktop`。

它应该依赖 SkillRun Core 的稳定 surface：

- CLI JSON contract
- registry/switchboard contract
- run record/envelope contract
- future daemon/router API

它不应该：

- 直接定义 Manifest schema。
- 直接执行 action。
- 绕过 `check`/Consumer Mode readiness。
- 直接信任 `.skr`。
- 从 MCP text content 反解析审计数据。
