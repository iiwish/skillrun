# SkillRun v0.5.3: Capsule Registry + Switchboard

**文档状态**：Ready_For_User_Review
**版本**：v0.5.3 planning baseline
**最后更新**：2026-05-15

---

## 一句话判断

v0.5.3 的核心不是“安装 skill”，而是给 v0.6 Router/Desktop 提供一个本地、可审查、可开关的 capsule inventory。

主题：

> **Capsule Registry + Switchboard: 先知道本机有哪些 capsule，再决定哪些允许被暴露。**

v0.5.3 不做 router、不改 MCP client 配置、不做 marketplace、不声称 trust。它只建立本地状态层：registered capsules 和 enabled/disabled 状态。

## 背景

v0.5.2 已经提供 `inspect/check/doctor --json`。这让外部控制面不再解析人类文本。

但 v0.6 仍然缺一个关键前置条件：

```text
哪些 capsule 在本机？
哪些被允许暴露？
哪些因为 Manifest stale 或 dependency 不满足而不能启用？
```

没有 registry，Router 只能扫描文件夹或依赖用户手写配置。没有 switchboard，Desktop 的开关只是 UI 假象。

## 核心模型

```text
Local Capsule Registry
  - capsule id
  - local path
  - source type
  - registered metadata
  - enabled flag

Switchboard
  - enable capsule
  - disable capsule
  - list enabled/disabled state
  - block enable when static checks fail
```

角色边界：

```text
.skr              = distribution artifact
registry          = local inventory and state
switchboard       = local exposure intent
router            = future MCP exposure layer
Desktop           = future UI over registry/router/run records
```

## 命令草案

v0.5.3 建议新增：

```bash
skillrun registry add --cwd <capsule> [--id <id>]
skillrun registry list --json
skillrun registry inspect <id> --json
skillrun registry remove <id>

skillrun switchboard list --json
skillrun switchboard enable <id>
skillrun switchboard disable <id>
```

默认输出可以保持人类文本，但 v0.5.3 的主合同应是 JSON，因为它服务 v0.6 headless control plane。

## Registry 数据边界

Registry 存储本地 inventory，不存储 trust 结论。

推荐字段：

```json
{
  "version": 1,
  "capsules": [
    {
      "id": "refund",
      "path": "D:/skills/refund",
      "source_type": "local_path",
      "enabled": false,
      "registered_at": "2026-05-15T00:00:00Z",
      "manifest": {
        "path": ".skillrun/manifest.generated.yaml",
        "present": true,
        "freshness": "fresh"
      },
      "skill": {
        "name": "refund"
      },
      "runtime": {
        "adapter": "python",
        "entrypoint": "action.py"
      },
      "tool": {
        "name": "refund"
      }
    }
  ]
}
```

建议实现约束：

- registry file 使用 `SKILLRUN_HOME` 覆盖；默认落在用户 home 下的 `.skillrun/registry.json`。
- registry 只记录 identity 和 state；readiness 可在 `list/inspect` 时实时计算。
- `registry remove` 只移除本地登记，不删除 capsule 文件。
- `registry add` 不执行 action source。
- `registry add` 可以登记有问题的 capsule，但默认 `enabled=false`。

## Switchboard 启用规则

`enabled=true` 的含义：

> 用户允许未来 Router 暴露这个 capsule。

它不表示：

- capsule 已被信任。
- action 被 sandbox。
- dependency 已被安装。
- MCP client 已经挂载。

`switchboard enable <id>` 应先跑静态/只读检查：

- Manifest 存在。
- Manifest freshness 为 fresh。
- `check` readiness 为 ok。
- capsule 没有 instruction-only 或 unsupported status。

如果检查失败，enable 失败并返回非零 exit code。用户仍可保留该 capsule 的 registry entry，但它不能进入 enabled set。

`switchboard disable <id>` 应总是尽量成功；它只改变本地 exposure intent。

## 非目标

v0.5.3 不做：

- 不导入 `.skr`。
- 不解包 `.skr`。
- 不做 SkillRun Router。
- 不暴露 MCP tools。
- 不修改 Claude Desktop、Cursor 或其他 MCP client 配置。
- 不做 local daemon。
- 不做 Tauri/Desktop。
- 不做 registry trust、签名验证或 marketplace。
- 不安装依赖。
- 不做 OS sandbox。
- 不改变 Manifest schema。
- 不改变 `run` / `test` envelope。

## 验收标准

- 可以登记一个本地 runnable capsule。
- 可以列出 registry，并输出单个 parseable JSON object。
- 可以 inspect registry entry，并复用 v0.5.2 的 readiness/inspect 信息。
- 默认新增 entry 为 disabled。
- enable 只在 Manifest fresh 且 readiness ok 时成功。
- stale Manifest、instruction-only、dependency-error capsule 不能 enable。
- disable 不执行 action source，也不删除 capsule 文件。
- 所有 registry/switchboard JSON 字段使用 snake_case。
