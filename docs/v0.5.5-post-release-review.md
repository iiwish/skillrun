# v0.5.5 发布后复盘

## 一句话判断

v0.5.5 是一次成功的内核合同硬化发布：它没有扩大 SkillRun 的产品边界，但把 Consumer Mode、Manifest schema、MCP 暴露、`.skr` 分发和 adapter lifecycle 统一到了更可信的运行合同上。

更重要的是，远端 Linux CI 在发布前暴露了本地 Windows 验证没有覆盖到的 Unix 进程组终止问题；这个问题被修复后才合并、打 tag 和发布，说明 v0.5.x 的质量闸门开始真正发挥作用。

## 发布事实

- 发布版本：`v0.5.5`
- 当前 binary/crate version：`0.5.5`
- Manifest IR `manifest_version`：仍为 `0.1.0`
- IPC / Adapter `protocol_version`：仍为 `0.1.0`
- Adapter Protocol：Level 0 command adapter 仍为 `adapter.v1`
- 发布分支：`codex/v0.5.5-core-contract-hardening`
- 合入主线：`main`
- GitHub Release：`https://github.com/iiwish/skillrun/releases/tag/v0.5.5`

发布验证：

- 本地 `cargo fmt --check` 通过。
- 本地 `cargo test` 通过。
- 本地 `cargo clippy --all-targets -- -D warnings` 通过。
- 本地 `git diff --check` 通过。
- 本地 `cargo run --quiet -- --version` 输出 `skillrun 0.5.5`。
- 远端 release branch CI 通过。
- 远端 main CI 通过。

## 这次发布真正完成了什么

v0.5.5 的核心不是“又多了几个命令”，而是把同一份运行事实收敛成更明确的系统边界。

已完成的关键收敛：

- `consumer::validate` 成为 Consumer Mode 进入执行、MCP 暴露和 pack 分发前的共享静态 Manifest gate。
- runtime 对缺失 `runtime.adapter` / `runtime.entrypoint` 的 Manifest fail-closed，并且不创建 run record。
- schema contract validation 前移到 Manifest generation、readiness、runtime、MCP dry-run 和 pack 路径。
- Core schema validator 覆盖当前承诺的 JSON Schema 子集，包括 `type`、`enum`、`required`、`properties`、`items`、`additionalProperties`、`minLength` 和 `minimum`。
- `ManifestView` 开始替代散落的 `serde_yaml::Value` 手写读取逻辑。
- adapter timeout lifecycle 抽到共享 process helper，Windows 使用 `taskkill /T /F`，Unix 使用 process group 的 `TERM` / `KILL`。
- release gate review 用命令矩阵明确了每个 consumer-facing command 的执行、暴露、分发边界。

## 这次暴露的问题

### P0: 本地验证不足以代表发布验证

本地 Windows 验证全部通过，但远端 Ubuntu CI 失败。失败点是：

`command_adapter_timeout_terminates_spawned_child_process`

根因是 Unix 下调用 `kill -TERM -<pgid>` / `kill -KILL -<pgid>` 时缺少 `--`，在 GNU `kill` 语义下负数进程组参数可能被解析为 option / signal，而不是进程组目标。

修复：

- Unix timeout cleanup 改为 `kill -TERM -- -<pgid>` 和 `kill -KILL -- -<pgid>`。
- 保留该测试作为跨平台 lifecycle 回归保护。

判断：

这是一个好失败。它证明 v0.5.x 的 hardening 方向是必要的，也证明 release branch CI 必须是合 main 和打 tag 前的硬闸门。

### P1: CI 失败可观测性不够

首次远端失败时，未认证 GitHub API 只能拿到 `Process completed with exit code 101`，拿不到完整 job log。

修复：

- CI 中的 `cargo test` 步骤改为显式 `tee cargo-test.log`。
- 失败时将末尾日志转成 GitHub annotation。
- annotation 输出做了截断和 ANSI 清理，避免关键信息被 HTML 或控制字符淹没。

判断：

这不是产品功能，但属于一流开源项目的基础设施质量。公开项目不能依赖维护者本地有 `gh` 或管理员 token 才能定位 CI 失败。

### P1: 发布状态文档需要最后一步回写

发布前 `RELEASE_NOTES.md` 的 v0.5.5 状态是 `Ready_For_Release_Decision`，发布完成后需要回写为 `Released`，并说明 main merge、tag、GitHub Release 已完成，package registry publication 未执行。

修复：

- v0.5.5 release notes 已回写 `Status: Released`。
- Publication 字段已注明 tag、GitHub Release 和 package publication 边界。

判断：

这一步应该进入固定 release checklist。否则文档会停在“准备发布”的状态，和真实 GitHub Release 事实不一致。

## 对项目定位的影响

v0.5.5 强化了“无护栏，不执行”的诚实边界。

这句话现在可以被更准确地解释为：

> SkillRun 不承诺 OS sandbox；它承诺在执行前用 Manifest、source hash、schema、preflight、permissions declaration、Consumer Mode diagnostics 和 structured envelope 把可审查合同立起来。

v0.5.5 支撑了其中三点：

- 可审计：run record、structured envelope、artifact、source hash 和 Consumer Mode static checks 有更一致的入口。
- 可验收：input/output schema 和 adapter output 不再只依赖 SDK wrapper，Core 也参与合同校验。
- 可复用：`.skr` 分发前必须过 Manifest 静态合同，减少坏包被打包传播的机会。

但仍然不能夸大：

- permissions 仍是声明、注入和审查边界，不是系统级权限隔离。
- adapter timeout cleanup 是 best-effort process lifecycle，不是恶意代码防护。
- `.skr` 不是 signed package、dependency bundle 或 reproducible runtime image。
- schema validator 是 SkillRun 支持的子集，不是完整 JSON Schema 引擎。

## 对 v0.6 Desktop 的影响

v0.5.5 让 Desktop 可以更放心地作为独立项目启动，但还没有让 Desktop 可以直接开始大规模 UI 开发。

已经足够稳定的输入：

- `inspect --json`
- `check --json`
- `doctor --json`
- registry / switchboard JSON
- run record 与 envelope
- Consumer Mode fail-closed 语义
- `.skr` source + Manifest archive 边界

仍需在 v0.5.6 或 v0.6 alpha 前补齐的输入：

- Router / daemon 的最小 headless contract。
- MCP client mount profile 的配置 diff / backup / rollback contract。
- Envelope Explorer 需要的 run record list/filter JSON surface。
- UI 不应读取 `.skillrun/` 内部目录作为稳定 API 的明确替代接口。

结论：

Desktop 可以单独建仓，但不应该先做漂亮 UI。下一步应该先把 headless consumer backend contract 冻住。

## 保留风险

### R001 Manifest reload contract 仍未定义

`serve --mcp` 启动时验证 Manifest，并使用启动时 snapshot。长连接期间 Manifest 或 source 改变不会自动 reload。

当前可接受，因为 v0.5.5 没有 daemon / hot reload 承诺。

进入 Router / daemon 后必须定义：

- reload 触发条件
- reload 前后 source hash 检查
- enabled capsule 是否需要重新 readiness gate
- 正在执行的 run 如何绑定 Manifest snapshot

### R002 Registry 是 inventory，不是 trust store

`registry add` 可以登记本地路径，但不表示 trust、install、enable 或 runnable certification。

当前可接受，但 Desktop UI 必须避免把“已登记”展示成“可信”。

### R003 Release automation 仍不够完整

当前发布仍依赖人工顺序执行：

- push branch
- inspect CI
- merge main
- rerun validation
- push main
- wait CI
- tag
- create GitHub Release
- update release notes status

这套流程已经可控，但仍容易遗漏“发布后回写文档”或“tag 前 main CI”。

## v0.5.6 推荐边界

v0.5.6 不应该继续扩张 runtime 能力。它应该是 release polish + headless contract preparation。

推荐范围：

1. 固化 release checklist
   - 把 v0.5.5 实际发布步骤写成 `docs/release-checklist.md`。
   - 明确 tag 前必须等待 main CI。
   - 明确 GitHub Release 后要回写 `RELEASE_NOTES.md`。

2. 固化 CI 诊断能力
   - 保留失败 annotation。
   - 可选：将 test failure tail 长度、ANSI 清理和 annotation 格式整理为稳定 CI convention。
   - 不引入复杂 CI matrix，除非已经有明确跨平台失败类型。

3. 设计 Router / daemon 最小 contract
   - 先写文档，不急实现。
   - 明确 Desktop 只能消费 CLI JSON / daemon API / run records，不读内部目录。
   - 定义 mount、registry、switchboard、run history 四类 API 的最小形态。

## 当时建议

1. 开 `v0.5.6` 小版本，只做发布工程、CI 可观测性和 headless consumer backend contract。
2. 先补 `docs/release-checklist.md`，把这次发布中真实踩过的步骤固定下来。
3. 再写 `docs/v0.5.6-headless-consumer-contract.md`，作为 Desktop 新仓库启动前的 API 边界。
