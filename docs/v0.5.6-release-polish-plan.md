# v0.5.6 Release Polish Plan

## 一句话判断

v0.5.6 应该是进入 v0.6 Desktop 之前的最后一个小而硬的 headless 版本：它不追求新场景，不做 UI，只把发布工程、CI 可观测性和消费者控制面 JSON contract 打磨到 Desktop 可以安全消费的程度。

## 版本主题

> Release polish and headless consumer contract.

中文：

> 发布工程固化与无界面消费者控制面合同。

## 核心取舍

v0.5.6 不应该变成“半个 Desktop”。

它应该只做三件事：

- 把 release checklist 变成维护者可以重复执行的流程。
- 把 CI 失败诊断变成默认能力，而不是发布当天临时补丁。
- 把 Desktop / Router 需要的 headless JSON surface 定义清楚，并只实现最小必要子集。

## 范围

### 必做

1. Release checklist 固化
   - `docs/release-checklist.md` 已建立。
   - 后续 release 必须明确 branch CI、main CI、tag、GitHub Release 和 release notes 回写顺序。

2. CI diagnostics 固化
   - 保留 `cargo test` 失败 annotation。
   - 确认失败 annotation 不影响成功路径。
   - 保持 `fmt`、`clippy`、`test` 三个 job 简洁，不引入过早 matrix。

3. Headless consumer contract
   - `docs/v0.5.6-headless-consumer-contract.md` 已建立。
   - 冻结 Desktop 不读 `.skillrun/` 内部目录、不解析 MCP text、不直接执行 action 的边界。

4. 最小可实现任务拆分
   - 先从已有 registry / switchboard JSON surface 做 inventory / exposure。
   - run history 和 mount plan 可以先定义 contract，再根据风险决定是否进入 v0.5.6 实现。

### 可选

- 为 `consumer inventory --json` 增加 alias 或新命令组。
- 为 `consumer exposure --json` 增加只读命令。
- 为 run records 增加只读 list/detail JSON。
- 为 MCP client mount 增加 plan-only JSON，不做 apply。

### 不做

- 不做 Tauri。
- 不做 `skillrun ui`。
- 不做 long-running daemon。
- 不修改 MCP client 配置。
- 不做 `.skr import` 的完整安装体验。
- 不做 signed package。
- 不做 OS sandbox。
- 不做 marketplace。

## 推荐任务拆分

### T001 Release Checklist Integration

目标：

- 确认 `docs/release-checklist.md` 与 `docs/release-policy.md` 不冲突。
- README / docs index 能找到 release checklist。

验收：

- `docs/README.md` 有入口。
- `docs/release-policy.md` 指向 checklist。
- `git diff --check` 通过。

状态：

- Done in `codex/v0.5.6-release-polish`。

### T002 CI Failure Diagnostics Review

目标：

- 复审 `.github/workflows/ci.yml` 中的 `cargo test` annotation。
- 确认成功路径输出仍然干净。
- 确认失败路径能暴露最后测试日志。

验收：

- 本地无法完整模拟 GitHub annotation 时，至少做 shell 语法审查和一次远端 CI 验证。
- CI 仍保留 `fmt`、`clippy`、`test` 三个 job。
- release checklist 记录该能力。

建议：

- v0.5.6 只 review 和保留，不继续复杂化。

### T003 Consumer Inventory JSON Contract

目标：

- 给 Desktop 一个稳定的本地 capsule inventory 数据源。
- 优先复用已有 registry / switchboard JSON。

候选命令：

```bash
skillrun consumer inventory --json
```

验收：

- 单个坏 Manifest 不拖垮整个列表。
- 每个条目包含 `id`、`path`、`enabled`、`readiness`、`skill`、`runtime`、`manifest`。
- 有 contract fixture 或集成测试。

风险：

- 不要让 `enabled=true` 被误读为 trusted/runnable。

状态：

- Done in `codex/v0.5.6-release-polish`。
- 已实现 `skillrun consumer inventory --json`，并用 contract fixture 与 registry 降级集成测试覆盖。

### T004 Consumer Exposure JSON Contract

目标：

- 明确 Router 未来应该暴露哪些 MCP tools。

候选命令：

```bash
skillrun consumer exposure --json
```

验收：

- `enabled=false` 不暴露。
- `readiness.ok=false` 不暴露。
- tool name 来自 Manifest-derived contract。
- 输出不执行 action。

风险：

- exposure 是未来 Router 的输入，不等于 Router 已实现。

状态：

- Done in `codex/v0.5.6-release-polish`。
- 已实现 `skillrun consumer exposure --json`，只暴露 `enabled=true` 且 `readiness.ok=true` 的 Manifest-derived tools。

### T005 Run History Read-Only Contract

目标：

- 给 Envelope Explorer 一个结构化 run history 输入。

候选命令：

```bash
skillrun consumer runs list --json
skillrun consumer runs inspect <run-id> --json
```

验收：

- list 默认不暴露完整 input。
- detail 绑定 `run_id`、capsule、Manifest hash、envelope、artifacts。
- 不把 stdout 当作成功结果。

风险：

- 当前 run records 分散在 capsule 目录下，索引策略要谨慎；不要为了 UI 便利把内部目录布局承诺成永久 API。

状态：

- Reviewed in `docs/v0.5.6-run-history-contract-review.md`。
- Done in `codex/v0.5.6-release-polish`。
- 已实现 registry-scoped `consumer runs list --json` summary，支持 `--capsule <id>` 与 `--limit <n>`。
- 不建议 v0.5.6 同时实现 `consumer runs inspect`、artifact content read、log content read、`--include-input` 或 global run index。

### T006 Mount Plan Contract

目标：

- 为“一键挂载”建立 plan-first 模型。

候选命令：

```bash
skillrun consumer mount plan --client claude-desktop --json
```

验收：

- 只输出将要修改什么。
- 不修改真实 MCP client 配置。
- 输出包含 `config_path`、`backup_path`、`changes`、`warnings`。

风险：

- apply / rollback 不建议进 v0.5.6，除非 mount plan 已经足够稳定。

状态：

- Reviewed in `docs/v0.5.6-mount-plan-contract-review.md`。
- Done in `codex/v0.5.6-release-polish`。
- 已实现 plan-only `consumer mount plan --client <id> --json`，支持 `--config <path>` 作为测试和高级用户 override。
- v0.5.6 不实现 `consumer mount apply`、`consumer mount rollback`，不修改真实 MCP client 配置。

## 推荐执行顺序

1. T002：先复审 CI 诊断，保证发布工程干净。
2. T003：实现或冻结 inventory JSON，因为 Desktop 的 Switchboard 第一屏依赖它。
3. T004：实现或冻结 exposure JSON，因为 Router 和一键挂载依赖它。
4. T005：设计 run history；是否实现视复杂度决定。
5. T006：只做 mount plan 设计，不做 apply。

## Definition of Done

v0.5.6 可以进入 release decision 的条件：

- release checklist 已进入 docs index。
- headless consumer contract 已进入 docs index。
- CI annotation 保留且远端 CI 通过。
- 若新增 consumer JSON 命令，必须有 fixture 或 contract test。
- README / release notes 明确 v0.5.6 没有 Desktop、daemon、OS sandbox、signed package 或 marketplace。
- `cargo fmt --check`、`cargo test`、`cargo clippy --all-targets -- -D warnings`、`git diff --check` 通过。
